# How 11 peer frameworks address pain points 02 and 04

Companion to [02-recompute-rewards-from-existing-rollouts.md](02-recompute-rewards-from-existing-rollouts.md) and
[04-judge-only-resume.md](04-judge-only-resume.md). Eleven products were analyzed at HEAD on 2026-07-07 —
the top-5 direct peers and top-5 eval harnesses by GitHub stars from the landscape survey, plus Inspect AI
(the adoption leader by installed use). Method: per product, one agent shallow-cloned the repo, read the two
pain-point docs for criteria, and investigated code + docs + issues; a second agent adversarially re-verified
every cited file/line/flag against the clone. Clones live under `/tmp/peer_pp_analysis/`.

Support scale: **first-class** (shipped, documented command) > **partial** (designed-for but with real holes)
> **workaround-possible** (technically scriptable, not a product feature) > **absent**.

## PP02 — re-score stored rollouts with a changed verifier, no re-inference

Grouped by the *method* each product uses; ordered strongest first. The last column carries the full flow: the base command that pays for inference, the re-score/recovery command, and the mechanism behind it.

| Method | Product | Support | <div style="min-width:520px">Flow: base run → re-score → mechanism & catch</div> | The command / hook | The catch |

| **A. Two-phase by design** — generation writes an artifact, scoring is a separate phase you can re-run | **Inspect AI** | ✅ first-class | **Base run** — `inspect eval task.py --model <provider/model>` — runs the eval end to end (inference + scoring) and writes `logs/<ts>_<task>.eval`, a self-contained log where every sample keeps its full message history, model output, target, and store. The catch: a change to the *evidence* stage (eval scripts, F2P/P2P test lists) still requires re-running containers (though still not the model), and `--rewrite_reports` requires the `test_output.txt` files to exist for every targeted instance. | `python -m swebench.harness.run_evaluation -p preds.jsonl --run_id <id>`; for grading-only changes add `--rewrite_reports True` — re-parses each stored per-instance test log (`test_output.txt`) into a fresh `report.json`, no containers spun up | verifier is two-stage: re-parsing/re-grading stored logs is free; changing eval scripts or the gold pass/fail test lists means re-running Docker containers (still zero model inference) |

## PP04 — judge-only resume after judge/scorer failures

| Method | Product | Support | <div style="min-width:520px">Flow: on failure → recovery → mechanism & catch</div> | Failure recording | Failed-subset re-scoring |
|---|---|---|---|---|---|
| **A. Structured per-sample error + failed-only retry** | **Inspect AI** | ⚠️ partial | **On failure** — the scorer exception is caught per sample and recorded as structured data: `EvalSample.error` carries message + traceback (plus an `error_retries` history), an ErrorEvent lands in the sample's event log, and the run continues; completed inference for that sample is already in the log.<br><br>**Recover** — `inspect eval-retry ./logs/<file>.eval` — re-runs only non-clean samples; samples with `error=None` are reused verbatim, their scores copied into the new run with zero re-inference and metrics recomputed over the merged set. Add `--checkpoint` on the *original* run to make retry scoring-only (`resume_for_scoring`). Alternative when all inference succeeded but judging was bad: `inspect score --action overwrite` re-judges everything from the log with no inference at all.<br><br>**Mechanism & catch.** Failure is a first-class per-sample state distinct from any score, which is what makes "re-run exactly the failed subset" definable at all. The catch: for an *errored* sample, `eval-retry` re-runs the whole sample — inference included — so the acceptance criterion "no new inference for failed samples" holds only via the checkpoint path or when failures didn't destroy stored output. | scorer exception → recorded on the per-sample log record as `EvalSample.error` (message + traceback); retried attempts land in the sibling `error_retries` field — stored as an error, never as a score | `inspect eval-retry log.eval` re-runs *only* non-clean samples (errored or invalidated) and copies clean samples with their scores verbatim (no re-inference or re-scoring for them); errored samples re-pay their own inference — close to, but not exactly, judge-only |
|  | **SWE-bench** | ✅ first-class | **On failure** — a scorer failure (docker build error, patch-apply failure, timeout — the scorer is deterministic test execution, no LLM judge) is caught per instance, logged with a full traceback to `run_instance.log`, counted in the progress stats, and manifests as the *absence* of that instance's `report.json`; the run continues.<br><br>**Recover** — re-run the *identical* command with the same `--run_id`: instances with a `report.json` are skipped ("N instances already run"), so exactly the failed subset re-scores, and aggregates are recomputed from the artifact tree — metrics come out as if the failures never happened.<br><br>**Mechanism & catch.** This meets PP04's acceptance criteria fully, but gets "no new inference" for free because generation lives entirely outside the harness (predictions file in, verdicts out). The catch: "absence of report.json" as the failure marker is fragile taxonomy — it cannot distinguish failure *kinds* without reading logs, and a crash that half-wrote artifacts needs manual cleanup. | failure = missing `report.json` (run summary counts it under `error_ids`) plus a traceback in that instance's `run_instance.log`; scoring is deterministic test-log parsing — no LLM judge anywhere | rerun with the same `--run_id`: instances that already have `report.json` are skipped; errored ones are fully re-executed (container re-runs the stored patch's tests) — zero model inference by construction |
| **B. Two-phase + judge checkpoint** | **OpenCompass** | ⚠️ partial | **On failure** — a soft judge failure (judge answered but no A/B verdict) becomes an `unknown` verdict via the postprocessor and scores 0; a hard failure (judge API dead after `retry` attempts, default 2) kills that eval task — but the predictions from the inference phase are safely on disk in the run dir.<br><br>**Recover** — `python run.py <config> -m eval -r <timestamp>` — re-runs *only* judging over stored predictions; the judge's own GenInferencer writes a `tmp_*.jsonl` checkpoint as it goes, so a re-run resumes judging mid-dataset instead of restarting it.<br><br>**Mechanism & catch.** This is genuine judge-only resume: policy inference is never re-paid because the eval phase never touches the policy model, and the judge checkpoint bounds re-judging cost after a mid-run crash. The catches: you must delete stale results files for datasets you want re-judged (the resume logic skips anything already scored), and `unknown` soft-failures stay conflated with legitimate wrong answers in headline metrics unless you mine the details files. | soft failure (judge reply lacks a parseable verdict letter) → recorded as `unknown` at verdict extraction; hard failure (judge API errors out after retries) → the eval task dies, but prediction files written by the inference stage survive | `-m eval -r <ts>` re-runs judging only, resuming mid-dataset from the judge's incremental checkpoint file (`tmp_*.jsonl`, saved every batch) — genuine judge-only resume; but `unknown` rows still conflate judge-said-nothing with genuinely bad output, both scoring as not-correct |
| **C. ## What this means for the `gym eval verify` RFC (synthesis across all 11)

1. **`inspect score` is the exact spec for `gym eval verify`** (PP02 fix hook): self-sufficient per-sample artifact + a command that re-runs scorers (with `--scorer`/args overrides) over stored logs and recomputes metrics into a merged log. Closest existing analogue to doc 02 §5.
2. **SWE-bench's two-tier verifier split** — persist intermediate *evidence* (its `test_output.txt` ≈ Gym's judge raw output in `judge_evaluations`) separately from the *grade*, so grading-logic changes replay from evidence for free.
3. **Per-sample durability before scoring** (Agent Lightning / Inspect): persist the response the moment inference completes, so a verify/judge failure can never destroy paid inference — Gym fix hook 04-§5.1 (`_ng_failure_class` in simple_agent) gets you this.
4. **Typed failure sidecar** (AgentBench `error.jsonl` + enum; Inspect `EvalSample.error`): Gym already has the `_failures.jsonl` sidecar and `_ng_failure_class` plumbing — only `stirrup_agent` uses it. Lifting it into `simple_agent` closes the biggest PP04 gap.
5. **The three-line skip** (OpenAI Evals `classify.py:64-66`): "if the response is already present in the row, skip the policy call" — the minimal-diff way to give Gym's simple_agent a verify-only path (doc 04 §5.2b proposes exactly this).
6. **Anti-patterns to avoid, observed in the wild:** silent judge-failure fallbacks (rLLM), swallowed groups (ART), unbounded blind retries (OpenAI Evals; slime's 401-retry), and a shipped-but-regressed offline scorer with no CI (verl `main_eval`) — argument for an e2e test on `gym eval verify` from day one.

Across the eleven: **no direct RL-training peer has first-class support for either pain point** — the mature answers all come from the eval-harness side (Inspect AI, OpenCompass, SWE-bench). A Gym that ships doc 02's `gym eval verify` + doc 04's failure sidecar would leapfrog every direct competitor on this axis.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `partial`**

verl ships a designed two-phase batch workflow: `main_generation_server.py` runs inference-only rollout over a prompt parquet and writes the same parquet back with a `responses` column (main_generation_server.py:183-189); `main_eval.py` then re-scores those stored responses by calling a reward function per row in Ray tasks — no model is loaded, no inference runs (main_eval.py:33-38) — and prints mean `test_score/<data_source>` aggregates (main_eval.py:72-76). Changing the verifier means editing/pointing `custom_reward_function.path` at a new file and re-running main_eval, so verifier iteration costs only scoring time. However this workflow is parallel to, not integrated with, RL training: in-training rewards are computed inside the reward loop and frozen; the trainer's rollout/validation JSONL dumps (`trainer.rollout_data_dir`/`validation_data_dir`, ray_trainer.py:454-477, 696-705) are not consumable by any shipped re-scoring tool. The in-trainer replay mechanism, SkipManager/RolloutSkip (`skip.rollout.action=cache`), reloads dumped generation DataProto and skips inference (rollout_skip.py:74-96), but in the default streaming-reward mode the cached batch already contains `rm_scores` (agent_loop.py:1067-1073) and the trainer only recomputes rewards when `use_rm` is set and `rm_scores` is absent (ray_trainer.py:1526), so cached replays reuse frozen rewards instead of re-verifying — re-scoring on replay happens only in colocate reward-model mode. Finally, main_eval on current main is regressed: `get_custom_reward_fn` reads `config.reward.custom_reward_function` (reward.py:70) while the stock `evaluation.yaml` still defines top-level `custom_reward_function` and no `reward` key, so the stock config crashes (independently reproduced via Hydra compose of the stock config: ConfigAttributeError "Key 'reward' is not in struct"), and the documented fallback to built-in scorers no longer exists in main_eval — a custom reward fn must be injected via Hydra `+` overrides (the override command was verified to compose and reach the custom fn).

```bash
Phase 1 (once, pays inference): python3 -m verl.trainer.main_generation_server data.train_files=test.parquet +data.output_path=gen.parquet actor_rollout_ref.model.path=<model> ... ; Phase 2 (repeat per verifier change, no inference): python3 -m verl.trainer.main_eval data.path=gen.parquet +reward.custom_reward_function.path=/path/my_reward.py +reward.custom_reward_function.name=compute_score  (Hydra + overrides verified to compose against the stock evaluation config; the reward function must be synchronous)
```

### PP04 — judge-only resume after scorer failures

**Support: `partial`**

**Failure recording:** Depends entirely on which reward manager is configured. Default `naive` reward-loop manager (default per verl/trainer/config/reward/reward.yaml reward_manager.name=naive): no try/except around compute_score (verl/experimental/reward_loop/reward_manager/naive.py:66-84) — a judge exception propagates through asyncio.gather (reward_loop.py:142) → ray.get (reward_loop.py:330) and crashes the step; nothing recorded. Legacy NaiveRewardManager catches only its own SIGALRM TimeoutError → reward 0.0 + console print (verl/workers/reward_manager/naive.py:129-143) — indistinguishable from a legitimate 0 in any artifact; the SIGALRM timeout additionally only works on the main thread and silently runs with no timeout otherwise (naive.py:38-52). PrimeRewardManager maps per-sample failure/timeout to None → 0.0 with only a print (prime.py:31-42,80-88) and zeroes the whole batch on a batch-level error (prime.py:143-148). Only the opt-in `rate_limited` manager (registered as 'rate_limited', limited.py:172) records distinguishable markers: reward=0.0 plus reward_extra_info['timeout']=True or reward_extra_info['error']=str(e) (verl/experimental/reward_loop/reward_manager/limited.py:451-467), which flow into rollout/validation dumps as extra columns — but the markers are fragile: batch collation takes extra keys from sample 0 only (reward_loop.py:345-349, agent_loop.py:1082-1085), so heterogeneous per-sample keys either KeyError or get silently dropped, and _write_generations drops any column whose length mismatches the batch (ray_trainer.py:468-470). So post-hoc failure-vs-low-score distinction is possible only in the rate_limited path, and unreliably.

**In-run retry:** Reward-model router calls (DisRM/GenRM): up to 16 attempts with exponential backoff min(2**attempt, 30)s; 4xx fail immediately, 5xx/timeout/connection/unexpected errors retry; after max retries the last exception re-raises into the reward manager (verl/experimental/reward_loop/reward_loop.py:157-195). The `rate_limited` manager adds LLM-as-judge protections: global concurrency semaphore, RPM and TPM token buckets (limited.py:420-428), and a per-sample asyncio.wait_for timeout defaulting to 300s (limited.py:365,430-438). Custom judge reward functions (the documented GenRM pattern, docs/advance/reward_loop.rst GenRM section ~lines 169-256) must implement their own retries — the doc example has none. The streaming per-sample judge call in the agent loop has no try/except (agent_loop.py:1000-1003).

**Resume:** Absent as a first-class feature — no way to re-run scoring for only failed samples. Search (grep -rniE 'rejudge|rescore|re_verify|reverify|retry_failed|failed_subset' over verl/ and docs/, re-run during verification) finds nothing; the only resume is checkpoint-based trainer resume (trainer.resume_mode, verl/trainer/config/ppo_trainer.yaml:162) at whole-training-step granularity, which re-runs generation. In training, a batch's rewards must succeed (or be zeroed) inside the step — there is no 'inference done, judge pending' state. The workaround lives in the offline flow: because inference is already frozen in the generation parquet, re-running main_eval after a judge outage re-scores with zero new policy inference and metrics are recomputed from scratch (trivially 'identical to a clean run') — but it re-pays ALL judge calls, not just failed ones, persists no per-sample scores to merge, and a single scorer exception kills the whole job with all scoring progress lost (main_eval.py:63-70, ray.get with no try/except).

```bash
python3 -m verl.trainer.main_eval data.path=gen.parquet +reward.custom_reward_function.path=/path/judge_reward.py +reward.custom_reward_function.name=compute_score  (whole-phase judge re-run over stored generations; offline flow only — no failed-subset selection, no command exists for the training flow; judge function must be synchronous because main_eval does not await async reward fns)
```

- `verl/experimental/reward_loop/reward_loop.py:157-195` — In-run retry for reward-model calls: 16 attempts, exponential backoff min(2**attempt,30)s, 4xx no-retry, re-raises last exception after exhaustion
- `verl/experimental/reward_loop/reward_manager/limited.py:451-467` — rate_limited manager converts judge timeout/exception to reward=0.0 with distinguishable reward_extra_info markers ('timeout': True / 'error': str(e)) instead of crashing

**Limitations:** The good failure behavior (markers + rate limiting + retries) is opt-in (`reward.reward_manager.name=rate_limited`) and not the default; the default path crashes the run exactly like NeMo Gym's simple_agent does, losing that step's completed inference. ## Agent Lightning — `microsoft/agent-lightning`

**Architecture:** Inference and scoring are fused inside the user's LitAgent rollout function, executed by LitAgentRunner (agentlightning/runner/agent.py:674-701): the agent makes its LLM calls (usually through the token-id-capturing LLMProxy, agentlightning/llm_proxy.py:149-175, AddReturnTokenIds) and then self-judges, either returning a float or calling emit_reward (agentlightning/emitter/reward.py:148), which persists the reward as just another OTel-style span. There is no separate verifier component; the LightningStore (agentlightning/store/base.py:104-124) durably records rollouts, attempts and all spans (including LLM prompts/responses and reward spans) during execution, and algorithms (VERL daemon agentlightning/verl/daemon.py, APO agentlightning/algorithm/apo/apo.py:390-427) later read spans back — APO extracts the last reward span via find_final_reward (agentlightning/emitter/reward.py:307), while the main VERL daemon takes the last non-None reward across adapter-produced triplets (verl/daemon.py:642-648).

### PP02 — re-score stored rollouts with a changed verifier

**Support: `workaround-possible`**

There is no re-scoring phase, command, or documented workflow: the reward is computed by user code inside the same rollout execution that pays for inference, and is frozen as a reward span in the store. However, the store's design accidentally enables post-hoc re-scoring by script: (1) the full trace — LLM prompts/responses, token IDs via the LLM proxy, tool calls — persists as queryable spans (query_spans); (2) add_span accepts new spans for any existing rollout/attempt, including finished ones (_add_many_spans_helper only verifies the rollout and attempt exist, it never checks terminal status); (3) span-level consumers take the LAST reward span (find_final_reward iterates reversed(spans)), so an appended re-scored reward span supersedes the frozen one for later APO/adapter/example passes (the main VERL path applies analogous last-reward-wins semantics but mediated by the TripletAdapter's reward-to-LLM-call matching, verl/daemon.py:642-648). So with a persistent store (Mongo backend or a long-running `agl store` server) a user can query each rollout's spans, recompute the reward from stored LLM outputs with the changed verifier, and append a new reward span via emit_reward(value, propagate=False) + Span.from_core_fields + store.add_span (the exact pattern the runner itself uses at runner/agent.py:294-313), or via the REST API documented in the OpenAPI spec (POST /v1/agl/spans, docs/assets/store-openapi.json). Nothing ships this: no CLI, no helper, no docs; the dashboard is strictly read-only (its only API layer, dashboard/src/features/rollouts/api.ts, contains exclusively method: 'GET' endpoints).

Evidence:

- `agentlightning/runner/agent.py (lines 674-692 inference, 294-313 reward-as-span, 701 find_final_reward)` — Inference and reward computation are fused in one rollout call; a float return is converted to a reward span (emit_reward at line 304, store.add_span at line 313) and stored — there is no standalone scoring phase
- `agentlightning/store/collection_based.py (lines 1149-1179, _add_many_spans_helper)` — add_span only verifies the rollout and attempt exist — it does not reject spans for finished (succeeded/failed) attempts, so post-hoc reward spans can be appended to stored rollouts (side effect: _post_add_spans also refreshes the attempt heartbeat)
- `agentlightning/emitter/reward.py (line 307, find_final_reward)` — find_final_reward returns the last reward span in sequence order (for span in reversed(spans)), so an appended re-scored reward span supersedes the original for its consumers: runner/agent.py:701, algorithm/apo/apo.py:417, contrib/agentlightning/contrib/algorithm/env_verl/daemon.py:484, examples/azure/aoai_finetune.py:285; the main VERL daemon instead takes the last non-None triplet reward via the adapter (verl/daemon.py:642-648), which is last-wins in spirit but mediated by reward-matching
- `agentlightning/store/client_server.py (line 784 POST /v1/agl/spans, line 832 POST /v1/agl/spans/next, line 810 POST /v1/agl/spans/search)` — The REST API (documented in docs/assets/store-openapi.json) exposes span query and span insertion, making the workaround possible from any process against a running store server
- `agentlightning/cli/store.py (lines 58-67, --backend {memory,mongo} and --mongo-uri) and docs/deep-dive/store.md (MongoLightningStore section, line 322)` — Persistence across runs requires the Mongo backend or a long-running `agl store` server; the default InMemoryLightningStore dies with the training process
- `dashboard/src/features/rollouts/api.ts (lines 256-389, all five endpoints method: 'GET'; no POST/PUT/DELETE anywhere in dashboard/src)` — The official dashboard is read-only — no re-run/re-score affordance exists in any shipped tool; grep for rescore/re-evaluate/recompute across the repo finds nothing (only VERL's unrelated 'recompute old_log_probs' at verl/trainer.py:311)
- `docs/tutorials/traces.md (Reading Traces section, ~line 101)` — Docs endorse post-hoc historical span analysis via LightningStore.query_spans combined with query_rollouts, but only for reading/inspection — never for re-writing rewards

**Limitations:** For the flagship RL path (VERL) this is largely moot: rollouts are consumed into a training batch live, per step, and on-policy training would regenerate rollouts anyway — post-hoc re-scoring cannot retroactively fix a finished training run. It is meaningful only for offline/analysis flows (APO history, the Azure SFT-from-traces example, custom eval scripts). The reward function is arbitrary user code inside the agent, so re-scoring from spans is only correct when the verifier is a pure function of the stored trace — judges that consult live environment state (DBs, sandboxes) cannot be replayed. The default in-memory store loses everything at process exit. And the entire path is undocumented tribal knowledge: users must reverse-engineer span semantics (reward span attribute layout, sequence-id allocation via get_next_span_sequence_id) to write the script. Additionally (verification pass): for the main VERL daemon the appended reward span only supersedes the original if the TripletAdapter's reward-matching strategy attaches it to the final triplet — last-wins is guaranteed only for find_final_reward-based consumers (APO, contrib env_verl, examples); and appending spans is not fully side-effect-free, since _post_add_spans refreshes the attempt heartbeat and can flip a 'preparing'/'unresponsive' attempt back to 'running' (collection_based.py:1203-1220), though terminal 'succeeded'/'failed' statuses are untouched.

### PP04 — judge-only resume after scorer failures

**Support: `partial`**

**Failure recording:** A judge exception inside the rollout is caught by LitAgentRunner._step_impl (agentlightning/runner/agent.py:710-714), which logs it and marks the attempt status='failed' (agent.py:727); iter() continues with the next rollout (agent.py:781) — the run does not crash. The failure IS distinguishable post-hoc at the control-plane level: rollout.status=='failed' (propagated via rollout_status_from_attempt, agentlightning/store/utils.py:60-84) vs a succeeded rollout with a low reward span, and query_rollouts(status_in=['failed']) (store/base.py:386-398) retrieves exactly the failed subset. Crucially, all spans emitted before the judge failed — including the completed inference LLM calls — are already persisted in the store, so the expensive half survives. Weaknesses: there is no error payload on the attempt (the Attempt model, types/core.py:136, has only status/worker_id/heartbeat/metadata — the traceback lives in runner logs), and at the metrics layer the distinction erodes — a rollout with no reward span yields final_reward=None, which the VERL daemon fills with reward_fillna_value=0.0 (agentlightning/verl/daemon.py:221, 1148-1149; it does log 'has_reward'/'n_rollouts_w_reward' separately at daemon.py:733-796 and training/n_rollouts_w_reward at 1098), while APO folds None into the average as 0.0 with `r or 0.0` (agentlightning/algorithm/apo/apo.py:502), conflating judge-never-answered with judged-wrong.

**In-run retry:** No judge-specific retry exists; the judge call is opaque user code (e.g., a bare OpenAI call in examples/apo/room_selector.py:94-119 with no try/except or backoff). The framework retry unit is the whole rollout: RolloutConfig(max_attempts, retry_condition) (agentlightning/types/core.py:161-172) — if the attempt's terminal status is in retry_condition and sequence_id < max_attempts, the store requeues the rollout (store/utils.py:70-82; requeue at store/collection_based.py:1527-1550) and a fresh attempt re-runs the ENTIRE rollout function, re-paying policy inference to retry the judge. Defaults are max_attempts=1, retry_condition=[] — by default a judge failure permanently fails the rollout with no retry. A watchdog additionally converts stalls to 'timeout'/'unresponsive' (scan_unhealthy_rollouts, store/utils.py:88-130) which can feed the same retry loop (docs/deep-dive/store.md, Rollout Transition Map, line 84). The store CLIENT retries network/5xx errors with backoff (store/client_server.py:1493-1547) but that covers store ops, not judge LLM calls; the LLM proxy's litellm num_retries (llm_proxy.py:1071) covers only policy calls routed through the proxy — and its default is 0, so even those are not retried unless configured.

**Resume:** Judge-only resume is not designed, but is scriptable through public store APIs because inference spans survive the failure: query_rollouts(status_in=['failed']) -> query_spans(rollout_id) to get the stored LLM outputs -> recompute the judge score externally -> start_attempt(rollout_id) (store/base.py:300-320, explicitly documented as 'Create a manual retry attempt for an existing rollout'; REST POST /v1/agl/rollouts/{id}/attempts, client_server.py:655) to open a fresh attempt without re-running the agent -> add_span with a reward span (client_server.py:784) -> update_attempt(status='succeeded') (client_server.py:676), which propagates the rollout to 'succeeded' via rollout_status_from_attempt. After that, any span-reading consumer (adapter, find_final_reward) sees the rollout as if it had succeeded, since they read the latest attempt/last reward span. Merging into metrics is where it breaks down: Agent Lightning has no run report — metrics are computed live per batch/round inside algorithms (VERL fills missing rewards with 0.0 into the already-consumed training batch, daemon.py:822-830; APO averages `r or 0.0` in the round it ran, apo.py:502) and pushed to wandb/console, so re-scored rollouts cannot flow back into a completed run's numbers; only a re-run of the aggregation over the store would reflect them, and no such re-aggregation tool ships.

Evidence:

- `agentlightning/runner/agent.py (lines 710-733, 781)` — Judge/reward exceptions are caught per-rollout (except at 710), the attempt is marked failed (727), and the iter() loop continues (781) — no crash, unlike NeMo Gym's dispatcher
- `agentlightning/types/core.py (lines 161-172) and agentlightning/store/utils.py (lines 60-84)` — In-run retry is whole-rollout only, gated by retry_condition/max_attempts; defaults (max_attempts=1 at line 168, retry_condition=[] at line 170) mean judge failures are not retried at all
- `agentlightning/verl/daemon.py (lines 221, 733-796, 1098, 1148-1149)` — Missing rewards are zero-filled into training/val metrics (reward_fillna_value=0.0) but tracked separately via has_reward/n_rollouts_w_reward — None-vs-0 partially distinguishable in logged metrics
- `agentlightning/algorithm/apo/apo.py (line 502)` — APO's average reward computes `r or 0.0` over finished rollouts, silently conflating judge-failure (None) with a legitimate 0 score
- `agentlightning/store/client_server.py (lines 655, 676, 784)` — Public REST/store APIs (start_attempt, update_attempt, add_span) make a scripted judge-only resume possible on top of surviving inference spans — but nothing ships it
- `https://github.com/microsoft/agent-lightning/issues/448` — Open, unanswered issue 'How to deal with failed rollouts?' (return None? auto-skip? batch abort?) — the failed-rollout recovery story is acknowledged-immature (verified open with no maintainer response)
- `https://github.com/microsoft/agent-lightning/pull/480` — Open PR 'Fix failed rollouts counted as completed in AgentModeDaemon' (fixes #355): failed rollouts (None reward, empty triplets from vLLM HTTP 400s) were counted as completed, producing 'Completed 33/32 tasks' and get_train_data_batch assertion failures; the fix adds _is_failed_rollout and skips them so whole-rollout retry re-claims them (full re-inference) — NOTE: this fix is NOT merged at the analyzed HEAD 0b40cb7 (_is_failed_rollout absent from the tree), so the miscounting bug is still live in AgentModeDaemon (verl/daemon.py:204)

**Limitations:** The comparison frame differs from Gym: Agent Lightning is a trainer, not an eval harness, so 'metrics identical to a clean run' has no natural target artifact — there is no aggregate-metrics file to regenerate, and for RL the batch that contained the zero-filled or skipped rollout has already updated the policy. The scripted resume path also requires the user to re-implement their judge outside the agent (the judge is embedded in the rollout function, so it cannot be invoked standalone), and requires a persistent store backend; with the default in-memory store, a crashed driver process loses everything including the inference spans. Additionally (verification pass): the LLM proxy's retry default is num_retries=0 (llm_proxy.py:1071), so out of the box not even policy inference calls are retried; and at the analyzed HEAD the PR #480 fix is unmerged, meaning the VERL daemon still counts judge/backend-failed rollouts as completed with fillna'd rewards rather than retrying them — the in-run failure-handling story is weaker at HEAD than the PR description suggests.

**Lessons for Gym:** Copy the durability model, not the scoring model: Agent Lightning streams every span — LLM calls, tool calls, rewards — into a central store DURING execution, so a judge failure never destroys completed inference and never kills the run (attempt marked 'failed', run continues); Gym's simple_agent could get the same property by persisting the response before /verify and stamping a failure class instead of raising. Second, an explicit per-rollout status machine ('failed' vs 'succeeded' plus a separate reward record) and VERL's has_reward/n_rollouts_w_reward metrics show how to keep infra-failure distinguishable from a legitimate zero at both the row and metrics layers — the standardized form of Gym's proposed verify_error field. Third, the append-only 'last reward span wins' semantics make re-scores naturally mergeable without rewriting history, a good model for gym eval verify output. Cautionary lesson: Agent Lightning fuses reward computation into the agent loop even harder than Gym does (the judge is arbitrary user code with no addressable /verify endpoint) and consequently ships NO re-score or judge-only-resume story at all — Gym's separated resources-server /verify endpoint is the better foundation; it just lacks the command on top.

## ART — `OpenPipe/ART`

**Architecture:** Inference happens in user-written rollout functions that call the policy through an OpenAI-compatible client served by the ART backend's vLLM (docs/fundamentals/training-loop.mdx; src/art/model.py:348-370 `openai_client`); scoring is architecturally separated: either the rollout fn assigns rewards itself, or the completed in-memory TrajectoryGroup is scored post-hoc by the `after_each=ruler_score_group` callback fused into `gather_trajectory_groups` (src/art/gather.py:40-44; src/art/rewards/ruler.py:266-368). Scored groups are persisted only at `model.log()` time (src/art/model.py:878) as per-step Parquet files under `{output_dir}/trajectories/{split}/{step:04d}.parquet` (src/art/model.py:954-958), with aggregate metrics appended to history.jsonl/W&B (src/art/model.py:704-726).

### PP02 — re-score stored rollouts with a changed verifier

**Support: `partial`**

ART's scorer is a pure post-hoc function over a completed TrajectoryGroup, which is genuinely 'designed for it' in memory: `ruler_score_group(group, judge_model, rubric=...)` deep-copies trajectories, preserves the prior reward in metrics['independent_reward'] (ruler.py:339), and replaces reward with the new judge score (ruler.py:363-365) — judge model and rubric are per-call arguments, not frozen run config, so re-scoring live groups with a changed judge/rubric costs zero policy inference. For STORED outputs the pieces exist but are unassembled: `model.log()` auto-writes every logged group (messages, rewards, metrics, metadata, tools, logs) to per-step Parquet (model.py:954-958), and the library ships `read_trajectory_groups_parquet` (trajectory_logging.py:153-251) whose output `ruler_score_group` accepts directly — so re-scoring a stored step is ~5 lines of Python against public APIs. However the reader is called only by unit tests (tests/unit/test_trajectory_parquet.py, tests/unit/test_frontend_logging.py) — nothing in src/ or examples/ uses it; even ART's own analytics loader (src/art/utils/benchmarking/load_trajectories.py) bypasses it and queries the parquet files directly via DuckDB for charting, never re-scoring. There is no CLI command (art CLI = install_skills|init|help|migrate|run, src/art/cli.py:58-230), no doc page, and no way to merge new scores back into history.jsonl/W&B. Exhaustive grep for rescore|re_score|rejudge|re_judge across src/, docs/, examples/ returns nothing.

Evidence:

- `src/art/rewards/ruler.py (lines 266-368)` — ruler_score_group re-scores an already-completed TrajectoryGroup with per-call judge_model/rubric; original reward preserved in metrics['independent_reward'] (line 339), reward replaced with judge score (lines 363-365) — scoring fully decoupled from generation
- `src/art/model.py (lines 954-958)` — model.log() (def at line 878) automatically persists trajectory groups to {output_dir}/trajectories/{split}/{step:04d}.parquet — a re-scoreable artifact written on every logged step
- `src/art/utils/trajectory_logging.py (lines 153-251)` — read_trajectory_groups_parquet reconstructs TrajectoryGroup objects from the parquet artifact, but nothing in src/ or examples/ calls it — only tests/unit/test_trajectory_parquet.py and tests/unit/test_frontend_logging.py (ART's own benchmarking loader src/art/utils/benchmarking/load_trajectories.py reads the parquet directly via DuckDB instead)
- `src/art/cli.py (lines 58-230)` — CLI surface is install_skills|init|help|migrate (JSONL-to-parquet format migration only, not re-scoring)|run — no re-score/re-verify command
- `src/art/utils/trajectory_logging.py (lines 46-56, 228-236)` — Parquet round-trip is lossy: _unflatten_message drops the trainable flag (messages come back as plain dicts, not Choice objects), logprobs/token IDs are never stored, and the reader reconstructs Trajectory without tools — re-read groups cannot be fed back to backend.train()

- `https://art.openpipe.ai/fundamentals/ruler` — Published docs show the 'Combining RULER with Independent Rewards / Adding Independent Rewards After Judging' pattern (verified via fetch) — re-scoring in-memory groups is the documented workflow; re-scoring from disk is never mentioned

**Limitations:** Only groups that reached model.log() are on disk — groups dropped by swallow_exceptions or trained-but-unlogged are gone. The parquet artifact is analysis-grade, not training-grade: no logprobs/token IDs, trainable flags dropped on read, tools dropped on read, so 'change judge, re-score, re-train on old rollouts' is impossible; only eval-style reward comparison works. There is no metrics-merge: history.jsonl/W&B rows are append-only per step and never recomputed, so re-scored rewards live outside the run's reporting. RULER is group-relative, so re-scored absolute values are only meaningful within a group. The stale docs claim about disk caching may mislead users into thinking rubric iteration is already cheap (third-party blog echo of that claim not independently verified). Additionally verified: even ART's internal analytics path (load_trajectories.py) treats the parquet as a raw DuckDB table rather than round-tripping through TrajectoryGroup, underscoring that the reader-to-rescorer path has zero production usage; group identity in the parquet is only a positional group_index per file, so cross-step or cross-run joining of re-scored results is on the user.

### PP04 — judge-only resume after scorer failures

**Support: `absent`**

**Failure recording:** A hard judge failure is recorded as ABSENCE, indistinguishable from data that never existed. With swallow_exceptions=True, ruler_score_group prints '[art_ruler] Swallowed exception: ...' to stdout and returns None (src/art/rewards/ruler.py:354-360); gather_trajectory_groups silently filters None out (src/art/gather.py:42-44, 55-63). The dropped group never reaches model.log(), so it appears in no parquet file and no metric — the exception_rate metric counts only rollout exceptions carried in group.exceptions (src/art/model.py:984-985), which a RULER failure bypasses entirely. In PipelineTrainer, a scoring exception inside rollout_fn increments a console-status 'errored' counter (src/art/pipeline_trainer/status.py:106-110, via trainer.py:518) and the scenario is consumed without requeue (src/art/pipeline_trainer/trainer.py:508-518). Soft failures (judge responds but malformed) raise ValueError/pydantic errors in ruler (ruler.py:230-231, 244, 249-252, 259-263) and follow the same drop path. A legitimate low score, by contrast, is a persisted reward + metrics['ruler_score'] + explanation log line (ruler.py:364-366) — so low-score and failure are technically distinguishable post-hoc, but only because failure leaves nothing at all, including no recoverable trajectories.

**In-run retry:** None in the RULER path. The judge call is a single litellm acompletion with caching=False and no num_retries (src/art/rewards/ruler.py:220-226); any exception propagates to ruler_score_group's except (ruler.py:354-360). With swallow_exceptions=False (the default, ruler.py:272) the exception escapes asyncio.gather in gather_trajectory_groups (after_each runs outside wrap_group_awaitable's try/except, gather.py:40-44) and crashes the training step. ART ships a generic exponential-backoff retry decorator (src/art/utils/retry.py:19-107) and even exports it publicly (src/art/__init__.py:86), but it has zero call sites in src/ — the megatron weight export uses its own hand-rolled _post_with_retry helper (src/art/megatron/weights/merged_weight_export.py:215), and ART's own examples reach for tenacity instead (examples/mcp-rl/mcp_rl/checks.py:8). Undocumented workaround: pass extra_litellm_params={'num_retries': N} to get LiteLLM-level retries, since kwargs are forwarded verbatim (ruler.py:225).

**Resume:** Impossible for the failed subset, by construction: the N completed policy trajectories in a judge-failed group exist only in memory and are discarded before any persistence, so there is nothing on disk to re-judge. Resume everywhere in ART is checkpoint-step-granular: example loops restart at `await model.get_step()` (examples/2048/train.py:44), and PipelineTrainer resumes by restoring scenario_offset and skipping consumed scenarios (src/art/pipeline_trainer/trainer.py:189-215, _skip_scenarios at 343-358) — full policy inference re-paid for any partially completed step; there is no 'inference done, judge pending' state. For groups that WERE persisted (logged pre-judge or with independent rewards), the PP02 parquet-scripting path could re-judge them, but no failed-subset marker exists to select on, and re-scored results cannot merge into history.jsonl/W&B metrics (append-only, never recomputed — src/art/model.py:704-726). Note the design rationale: for GRPO training, dropping a judge-failed group is semi-defensible since the next step generates fresh on-policy data anyway; but the same drop applies in eval/benchmark use of RULER (examples/mcp-rl/mcp_rl/benchmarks/generate_benchmarks.py:64-69), where lost samples are simply missing data.

Evidence:

- `src/art/rewards/ruler.py (lines 354-360)` — Judge failure handling is print-and-return-None (swallow_exceptions=True) or raise (default, crashes the gather) — no retry, no persisted failure record
- `src/art/gather.py (lines 40-44, 55-63)` — after_each is fused into gather; None results from failed judging are silently filtered out of the returned groups, so all completed policy trajectories in that group are discarded
- `src/art/rewards/ruler.py (lines 220-226)` — Single acompletion call, caching=False, no num_retries — the only retry knob is the undocumented extra_litellm_params passthrough
- `src/art/utils/retry.py (lines 19-107)` — A generic retry decorator with exponential backoff exists and is exported from art.__init__ (line 86), but has zero call sites in src/ — it is not applied to judge/scorer calls; megatron weight export uses a separate hand-rolled _post_with_retry (merged_weight_export.py:215) and examples use tenacity
- `src/art/model.py (lines 984-985)` — exception_rate metric counts only group.exceptions from rollout failures; RULER-swallowed groups never reach model.log so they are invisible even to this metric
- `src/art/pipeline_trainer/trainer.py (lines 189-215, 343-358, 508-518)` — PipelineTrainer resume=True restores scenario_offset/step and skips consumed scenarios; a scoring failure inside rollout_fn prints, marks a console 'errored' counter (status.py:106-110), and permanently consumes the scenario — no requeue, no judge-only retry
- `docs/fundamentals/ruler.mdx (lines 273-282) + https://art.openpipe.ai/fundamentals/ruler` — Official docs recommend swallow_exceptions=True as the production pattern: 'groups that fail to be judged are simply filtered out rather than crashing the training loop' (verified in both local mdx and published page) — data loss is the documented answer to judge failure
- `https://github.com/OpenPipe/ART/issues/476` — Real-world report of RULER judge responses failing schema validation (Ollama /api/chat judge cannot produce RULER's required JSON schema; verified via fetch), i.e. soft judge failures occur in practice and hit this drop path

**Limitations:** Verified absences: exhaustive grep for rescore/rejudge/re_judge in src/, docs/, examples/ returns nothing; the retry decorator has zero call sites in the reward path (or anywhere in src/). UNVERIFIED: the claim that GitHub issue search surfaced no RULER retry/rescore/resume feature or roadmap item could not be re-run (API rate-limited); only issue #476 was directly confirmed. Caveats to the severity: (a) unlike a crash-on-failure design, ART's docs-recommended swallow_exceptions=True degrades gracefully, so a judge outage costs data, not the job — but note the code DEFAULT is swallow_exceptions=False (ruler.py:272), so out-of-the-box a judge failure crashes the training step unless the user follows the docs; (b) because RULER scores groups relatively, per-sample judge retry is not meaningful — the retry unit would be a whole group, which slightly reshapes what 'judge-only resume' should mean here. Additional miss: a swallowed judge failure also bypasses gather's own max_exceptions accounting (gather.py:152-156 counts only exceptions raised from the rollout awaitable), so even the in-run exception budget never sees judge failures.

**Lessons for Gym:** For PP02, copy ART's scorer contract: `ruler_score_group` is a pure function over a completed trajectory artifact whose judge/model/rubric are per-call parameters (not config frozen at server start), and it preserves the old reward in `metrics["independent_reward"]` for before/after comparison — a `gym eval verify` should likewise take verifier overrides per invocation and stamp the prior reward into re-scored rows. Also copy automatic, self-contained trajectory persistence at log time (per-step Parquet with messages+rewards+metadata), but fix ART's mistake of a lossy reader — Gym's rollouts.jsonl should stay a lossless superset of the verify request. For PP04, ART is mostly an anti-lesson: swallow-and-drop (return None, filter group) destroys paid inference with no persisted marker and hides failures from every metric including its own exception_rate, which validates a persist-plus-failure-class sidecar as the strictly better design; the one thing worth copying is that (when configured per docs) scoring failures never kill the run.

## slime — `THUDM/slime`

**Architecture:** Policy inference runs on SGLang engines managed by a Ray RolloutManager actor (slime/ray/rollout.py:409-547 starts servers and drives rollouts); each rollout task POSTs to the SGLang router in generate() (slime/rollout/sglang_rollout.py:153-220). Scoring is fused into the tail of the same async task: generate_and_rm() calls rm_hub.async_rm/batched_async_rm immediately after generation (slime/rollout/sglang_rollout.py:263-286), dispatching to rule-based graders, a remote RM over HTTP, or a user function via --custom-rm-path (slime/rollout/rm_hub/__init__.py:55-110). Rewards are frozen onto Sample.reward and consumed by _post_process_rewards/_convert_samples_to_train_data in the RolloutManager (slime/ray/rollout.py:664-739); there is no separate verification service or verify endpoint — scoring is an in-process function call inside the rollout phase.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `workaround-possible`**

slime cleanly separates the rollout phase from the training phase via disk artifacts: --save-debug-rollout-data (or --dump-details, which auto-sets it at slime/utils/arguments.py:1837-1839) serializes every Sample — prompt, full token trajectory, response text, label, metadata, and reward — to a per-rollout-step .pt file (slime/ray/rollout.py:645-662), and --load-debug-rollout-data replays those files into training with SGLang never even started (arguments.py:1841-1846 auto-sets debug_train_only, so zero policy inference). However, the shipped replay path reuses the frozen sample.reward and never re-invokes the reward model: _get_rollout_data just does Sample.from_dict and hands samples to train-data conversion (slime/ray/rollout.py:614-643), and the alternate forge_load replay path likewise returns stored rewards untouched (slime/rollout/forge_load.py:100-114). Re-scoring with a changed verifier is therefore only achievable through user code: either (a) hook --custom-reward-post-process-path, which replaces _post_process_rewards and receives the full loaded samples, so it can recompute rewards from sample.response/sample.label before advantage computation (slime/ray/rollout.py:664-666); or (b) edit the dumps offline — batched_async_rm(args, samples) is a pure function of the dump contents plus args (slime/rollout/rm_hub/__init__.py:99-110), so a ~15-line script can load the .pt, re-run it with a new --custom-rm-path/--rm-type, overwrite sample.reward, and re-save for replay. Nothing ships or documents either recipe; the docs frame the dump/replay flags purely as debugging aids for the training side (docs/en/developer_guide/debug.md:45-51).

```bash
No shipped re-score command (verified absent: repo-wide grep for rescore|re-score|reverify|rejudge finds nothing). Workaround (all flags verified in slime/utils/arguments.py:1239,1360,1541,1543): phase 1 collect once with `python train.py <cfg> --debug-rollout-only --save-debug-rollout-data '/dumps/{rollout_id}.pt'`; phase 2 re-score + train with zero inference via `python train.py <cfg> --load-debug-rollout-data '/dumps/{rollout_id}.pt' --custom-reward-post-process-path my_rescore.rescore`, where user-written rescore(args, samples) recomputes rewards from the stored sample.response/sample.label and returns (raw_rewards, rewards) (alternatively rewrite the .pt dumps offline with slime.rollout.rm_hub.batched_async_rm and replay unmodified).
```

**Limitations:** Only works if the user turned on --save-debug-rollout-data before the run; dumps are torch pickles (loaded with weights_only=False at slime/ray/rollout.py:615-617) keyed per rollout step, so re-scoring a whole run means iterating many files. The replay path deliberately skips rollout metric logging (slime/ray/rollout.py:1279-1280), so re-scored rewards flow into training but not into wandb rollout metrics. Eval results have no shipped re-ingest path — eval rewards are aggregated and logged in-process (slime/ray/rollout.py:1240-1268), so changed-judge eval numbers require a full eval re-run (forge_load's eval branch, forge_load.py:72-91, can technically replay an eval dump into _log_eval_rollout_data, but with frozen rewards and SGLang servers kept alive — it is a memory-testing tool, not a re-scorer). Stateful verifiers (e.g. examples/coding_agent_rl's sandbox-based SWE evaluation) cannot be re-run from a dump alone. The post-process hook is synchronous (called without await at rollout.py:666), so LLM-judge re-scoring inside it must manage its own event loop. MISSED BY ORIGINAL: the --custom-reward-post-process-path hook replaces _post_process_rewards wholesale, so a re-scoring hook must also reimplement the default GRPO/GSPO group-mean/std reward normalization (slime/ray/rollout.py:668-689) or training advantages will silently differ from a normal run; and dumps are version-fragile — Sample.from_dict drops unknown pickled fields into setattr and torch.save pickling ties dumps to compatible slime/torch versions, so long-lived rollout archives are not a stable re-scoring substrate.

### PP04 — judge-only resume after scorer failures

**Support: `absent`**

**Failure recording:** In-core: a hard scorer failure is not recorded at all — it is an unhandled exception. remote_rm re-raises after exhausting retries (slime/rollout/rm_hub/__init__.py:46-49) and no layer above catches it (generate_and_rm has no try/except around async_rm at sglang_rollout.py:282-284; call_rollout_fn in slime/rollout/base_types.py:19-26 passes exceptions through), so the sample (and its paid inference) exists only in memory and dies with the process. A Sample.Status.FAILED value exists in the schema, documented as 'recoverable or non-critical failure... external API error' (slime/utils/types.py:130-138), but repo-wide grep confirms it is never assigned anywhere — it is dead schema. The only distinguishable-failure convention lives in user land: examples/coding_agent_rl coerces any rollout/scorer exception to reward=0.0 + Status.ABORTED + metadata['abort_reason'] + remove_sample=True (examples/coding_agent_rl/generate.py:316-333, invoked from exception handlers at :272,:280), which is post-hoc distinguishable from a legitimate 0 via metadata and excluded from loss via the zeroed mask — but in eval aggregation rewards are averaged blindly (slime/ray/rollout.py:1247-1249), so a judge failure and a wrong answer are indistinguishable in eval metrics.

**In-run retry:** Two in-run retry layers, both generation-era: remote_rm retries up to 10 attempts with exponential backoff min(2^n,30)s + jitter on ANY exception (bare `except Exception` — no status-code discrimination, a 401 retries as eagerly as a 429), then raises (slime/rollout/rm_hub/__init__.py:34-52); the shared HTTP helper used for SGLang generation retries 60 times at 1s intervals (slime/utils/http_utils.py:165-198). Rule-based RMs are local pure functions (no retry needed); --custom-rm-path functions are entirely on their own for retries.

**Resume:** None — judge-only resume is impossible by construction. Default sync path: an RM exception propagates through asyncio.gather in generate_and_rm_group into task.result() at slime/rollout/sglang_rollout.py:417 (train) or `await coro` at :589 (eval), up through RolloutManager.generate, and crashes train.py at the ray.get on line 53; every in-flight sample is lost because the data buffer is in-memory only and DataSource.save() persists just dataset cursors, never samples (slime/rollout/data_source.py:123-136,171). Checkpoint resume restarts at whole-rollout-step granularity (start_rollout_id = loaded_rollout_id + 1, slime/backends/megatron_utils/actor.py:100), regenerating all inference. The fully-async path survives instead of crashing, but by silently dropping the crashed group and generating a fresh one — new inference, sample lost (slime/rollout/fully_async_rollout.py:169-181; only ABORTED groups are requeued at :183-188, exception groups never are). The one internal seam that resembles 'inference done, scoring pending' — generate_and_rm recomputes the RM iff sample.reward is None (sglang_rollout.py:272-284) — never reaches disk, and a COMPLETED sample with reward=None actually trips an assert when group_rm is off (sglang_rollout.py:235-238). Dumps can't rescue a crashed step either: _save_debug_rollout_data runs only after the whole rollout step returns (slime/ray/rollout.py:540-541), so a scorer crash means no dump was ever written. No merge-back into metrics exists because no failed-subset artifact exists.

Evidence:

- `slime/rollout/rm_hub/__init__.py:34-52` — remote_rm: 10 retries with capped exponential backoff on any exception, then raise — the only scorer retry policy in core
- `slime/rollout/sglang_rollout.py:415-417` — task.result() in the rollout collection loop has no try/except; an RM exception kills the entire rollout step
- `train.py:53` — ray.get(rollout_manager.generate.remote(rollout_id)) is unguarded — the RM exception crashes the whole training job
- `slime/utils/types.py:130-138` — Status.FAILED is defined for exactly this case ('external API error') but grep shows zero assignments repo-wide — the failure marker is unwired
- `slime/rollout/fully_async_rollout.py:169-181` — fully-async done-callback logs 'process task raised' and returns — the group with completed inference is silently discarded, not requeued for scoring
- `slime/rollout/sglang_rollout.py:235-238,272-284` — reward recomputed only when sample.reward is None; a COMPLETED sample without a reward asserts (when group_rm is off) — no persisted 'scored-pending' state can exist
- `slime/rollout/data_source.py:123-136` — buffer save() persists only sample_offset/epoch_id/index counters; buffered samples with completed inference are memory-only (buffer initialized as plain list at data_source.py:171) and lost on crash
- `examples/coding_agent_rl/generate.py:316-333` — ecosystem workaround pattern: coerce scorer exceptions to reward=0 + metadata['abort_reason'] + remove_sample=True — user code, not framework
- `docs/en/advanced/fault-tolerance.md:11-21` — fault tolerance scope is explicitly rollout-ENGINE health (restart SGLang servers); reward/judge-call failure recovery is out of scope, deferred to 'checkpoints and debug dumps'

**Limitations:** The absence is structural: scoring runs inside the same asyncio task as generation with persistence only after the full rollout step succeeds, so a hard judge failure destroys the inference it was scoring. Soft failures (user RM returns 0 with a metadata marker) are recoverable for training rewards via the PP02 dump-editing workaround, but only if dumps were enabled and the step completed; eval metrics can never be patched post-hoc. Everything distinguishable about a failure depends on conventions the user implements in their own custom RM / generate function. MISSED BY ORIGINAL: the --group-rm path has identical crash semantics — the whole-group batched_async_rm call after gather (sglang_rollout.py:327-331) is equally unguarded, so one judge failure destroys the inference of the entire n_samples_per_prompt group at once; and the partial-rollout abort machinery (sglang_rollout.py:336-372) — the only mechanism that persists completed inference back into the buffer mid-step — is triggered only by deliberate aborts, never by scorer exceptions, underscoring that requeue-on-failure was buildable but not built.

**Lessons for Gym:** For PP02, slime's artifact design is the lesson: one self-describing record per sample (inputs + trajectory + grading fields + reward via Sample.to_dict) plus a first-class replay flag whose mere presence disables inference (--load-debug-rollout-data auto-sets debug_train_only) — Gym could adopt both by making a rollouts.jsonl row a complete, standalone verify request (killing the materialized-inputs join) and shipping `gym eval verify` as the analogous inference-off replay mode; slime's gap (replay reuses frozen rewards, and its one re-score hook forces users to also reimplement advantage normalization) is exactly the hole Gym's proposed command would fill, so Gym can leapfrog rather than copy. For PP04, slime is a cautionary tale, not a model: it too fuses scoring into the generation task, persists nothing until the step succeeds, and its FAILED status enum is defined but never set — evidence that a failure marker only matters if the framework itself sets it and routes marked rows to a resumable artifact, which is precisely Gym's proposed _ng_failure_class sidecar + verify-only dispatch plan.

## rLLM — `rllm-org/rllm`

**Architecture:** One engine, AgentFlowEngine, runs both training and eval: inference happens inside the agent flow against the model gateway in `_run_flow_only` (rllm/engine/agentflow_engine.py:575-644), and scoring happens immediately afterwards, in-process and in the same per-task pipeline, via `ctx.evaluator.evaluate(task, enriched_episode)` in `_finish_episode` (rllm/engine/agentflow_engine.py:678-682). Verifiers are host-side Evaluator objects — a Protocol `evaluate(task, episode) -> EvalOutput` (rllm/types.py:493-501) — resolved per task by SandboxTaskHooks (rllm/hooks.py:201-291); sandbox-backed verifiers (tests/test.sh) run inside a per-task ephemeral sandbox (rllm/eval/script_evaluator.py:55-107). `rllm eval` persists one full Episode JSON per rollout by default plus an aggregate results.json (rllm/eval/episode_store.py, rllm/eval/results.py, rllm/cli/eval.py:380-405,446-455).

### PP02 — re-score stored rollouts with a changed verifier

**Support: `workaround-possible`**

At run time generation and scoring are fused: the evaluator runs inline at the tail of the same per-task pipeline that did inference (agentflow_engine.py:526-561, 678-682). However, `rllm eval` persists a complete Episode JSON per rollout BY DEFAULT (`--save-episodes` default true): trajectories, steps with chat_completions/model_response, artifacts, and the full task metadata are all in one self-contained file under `<run_dir>/episodes/` (episode_store.py:1-18, 96-106). Because the Evaluator contract is a pure host-side function of `(task, episode)` (types.py:493-501) and evaluators are loadable by name/import-path via `load_evaluator()` (evaluator_loader.py:231-263), re-scoring stored episodes with a changed verifier is a short hand-rolled script: load each episode JSON, `Episode.model_validate()`/`Episode.from_dict()` it (types.py:317, 369-380), call the new evaluator, and rebuild metrics with `EvalResult.from_items()` (results.py:64-110). Nothing in the product ships or documents this: the full CLI is agent/dataset/eval/init/model/sft/snapshot/train/view/login plus a hidden `setup` alias (cli/main.py:29-42); the only consumers of saved episodes are `rllm dataset from-eval` (SFT curation — reads the FROZEN reward/is_correct from results.json, never re-scores; curation.py:1-18) and `rllm view` (read-only browser, zero evaluator references in visualizer.py/view.py). The `--evaluator` override flag exists but only for a full run that re-pays inference (cli/eval.py:541).

The only shipped knob is `rllm eval <bench> --evaluator <name>` which re-runs inference.

### PP04 — judge-only resume after scorer failures

**Support: `workaround-possible`**

**Failure recording:** Three distinct behaviors, none surfacing in headline metrics: (1) The built-in LLM-judge reward fns swallow ALL judge exceptions and silently substitute a different scoring rule — `_call_judge` catches `Exception`, logs a warning, returns None (llm_judge.py:108-110; llm_equality.py:125-127); llm_judge's fallback then awards reward=1.0 for ANY non-empty answer with metadata `{"reason": "judge_unavailable_fallback"}` (llm_judge.py:71-79) — a judge outage INFLATES scores; llm_equality falls back to token-F1 with `{"method": "f1_fallback"}` (llm_equality.py:68-75). (2) Python module verifier exceptions → reward=0.0 + metadata `{"error": "verifier exception: ..."}` (module_evaluator.py:78-81); shell verifier missing reward file → reward=0.0 + `{"error": "no reward file found"}` (script_evaluator.py:133-134). (3) A custom evaluator that raises through → in eval, an error Episode with `termination_reason=ERROR`, metadata.error, and the trajectories DROPPED — the completed inference is lost (agentflow_engine.py:511-523). Post-hoc distinguishability: EvalOutput.metadata is merged into `episode.metrics` (agentflow_engine.py:700) and persisted in the per-episode JSON, so cases (1)/(2) ARE detectable by scanning episode files; but results.json items carry only float signals and an error string reserved for crashed episodes (runner.py:154-183; results.py:13-25), so aggregate accuracy silently includes fallback-scored samples with no marker — and the fallback even writes a judge_score Signal that flows into results.json signal_averages, inflating those too. Recency note: PR #710 ('Classify infra/grading failures distinctly across eval + train', merged 2026-07-02) adds typed GRADING_ERROR/VERIFIER_TIMEOUT/SANDBOX_ERROR (plus AGENT_SETUP_TIMEOUT/ENV_START_TIMEOUT/MODEL_ERROR) TerminationReasons and per-reason training masks (CompactFilteringConfig) — but into the `terminal-rl` feature branch, NOT main; main's enum still has only 7 generic reasons (workflows/workflow.py:18-26).

**In-run retry:** No rLLM-level retry exists around judge/scorer calls. The judge fns construct a plain `OpenAI(base_url=..., api_key="EMPTY")` client with no max_retries argument (llm_judge.py:86-98), so only the OpenAI SDK's implicit default (max_retries=2 on connect errors/429/5xx) applies — incidental, not designed. The engine's only retry is WHOLE-ROLLOUT: `process_task_with_retry` re-runs flow + trace fetch + enrich + evaluate on any exception (agentflow_engine.py:458-524). Training uses config retry_limit (default 3 in rllm/trainer/config/rllm/base.yaml:37 and agent_ppo_trainer.yaml:44) with raise_on_error defaulting True (unified_trainer.py:211-212) — an evaluator that raises burns retry_limit full inferences then crashes the step; eval hardcodes retry_limit=1, raise_on_error=False (eval/runner.py:107-110) so the run completes and the failure becomes an error Episode counted in `result.errors`.

**Resume:** No judge-only resume exists. The documented recovery is `rllm eval <bench> --task-indices 4,11,27` — the docs call this 'Great for re-running a handful of failures' (docs/datasets/running-evaluations.mdx:141-143, example at ~173-176) — which re-dispatches those tasks through the FULL pipeline (new inference, new sandbox) and writes a NEW timestamped run dir with its own results.json; EvalResult has save/load/from_items but no merge function (results.py:124,162), so re-run results never fold back into the original run's metrics (curation pools runs by task id in _build_groups, but only to emit SFT rows, not metrics). Workaround for soft judge failures: because episodes persist with everything host-side judges read (episode.artifacts['conversation'] read at llm_judge.py:43-46; trajectory-output fallback chain in reward_fns/_helpers.py:8-23) and the fallback marker survives in episode JSON metrics, a hand-rolled script can filter episodes where `metrics.reason == "judge_unavailable_fallback"`, re-call the judge evaluator offline (same recipe as PP02), and rebuild EvalResult.from_items — metrics then match a clean run since score is a pure function of items. This does NOT work for hard evaluator exceptions in eval: the error Episode is persisted WITHOUT trajectories (agentflow_engine.py:513-523), so the inference is unrecoverable, exactly Gym's PP04 failure mode.

```bash

```

Sandbox verifier failures (timeout, dead container) are additionally unrecoverable offline since container state is not persisted; script_evaluator deliberately logs non-zero verifier exits at debug level only (script_evaluator.py:91-102), so a systematically broken verifier script is invisible in logs too. [Verifier addition] The eval on_episode_complete callback swallows its own exceptions at debug level (runner.py:167-172), so the audit trail (episode JSONs) needed for the workaround can itself be silently incomplete.

**Lessons for Gym:** Copy the artifact design, not the scoring pipeline: rLLM saves one self-contained Episode JSON per rollout by default, with the verifier's inputs (task metadata, conversation artifacts, trajectory outputs) embedded alongside the model output — eliminating Gym's lossy rollouts.jsonl/materialized-inputs two-file join that PP02 §4 flags, and its evaluator-as-pure-function-of-(task, episode) Protocol (rllm/types.py:493-501) is the property Gym should formalize as a `supports_offline_verify` flag. From the PR #710 taxonomy (typed GRADING_ERROR/VERIFIER_TIMEOUT/SANDBOX_ERROR termination reasons with per-reason masking of training data — verified merged, though only into the `terminal-rl` branch), Gym should copy the idea of a schema-level verify-failure class distinct from reward=0 — exactly PP04 §5's `_ng_failure_class="verify"`/`verify_error` proposal. Two cautionary anti-patterns to avoid, both verified in code: rLLM's judge fns silently substitute a different scorer on failure (reward=1.0 for any non-empty answer when the judge is down, llm_judge.py:71-79), which is worse than crashing; and its eval error path persists episodes with trajectories stripped (agentflow_engine.py:513-523), losing paid inference the same way Gym's simple_agent does. Notably, even this well-funded direct peer ships NO re-score or judge-only-resume command — its own design doc (design/sft-distillation.md:35-40) admits users must hand-write episode-replay scripts — so a first-class `gym eval verify` would be a genuine differentiator, not table stakes.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `partial`**

Re-scoring rides on the `--use_cache <prefix>` SQLite response cache, not on the human-readable sample logs. CachingLM wraps the model and keys each response by sha256 of json.dumps([request_type] + request args) — i.e. [context_string, gen_kwargs] for generate_until (lm_eval/api/model.py:230-232, key computed at :284); metrics, filters, and process_results are NOT in the key. So the user edits the task YAML's filter/metric/process_results (or the judge prompt inside a process_results function) and re-runs the identical eval command with the same cache path: every loglikelihood/generate_until call is a cache hit (model.py:283-308), zero model forward passes happen, and filters + process_results + aggregation recompute fresh scores.
```bash
lm-eval run --model hf --model_args pretrained=<M> --tasks <task> --use_cache /path/cache --output_path results/ --log_samples   # collect once; then edit the task YAML's filter/metric/process_results and re-run the IDENTICAL command — all model calls hit the sqlite cache, only scoring recomputes  [all flags verified: run.py:112-117 (--use_cache), :170-183 (--output_path/--log_samples)]
```

Also, --use_cache DBs are per-rank, so crash recovery additionally requires re-running at the same world size (evaluator.py:276-289, README.md:648).

**Lessons for Gym:** Two patterns transfer directly, and both survived verification. (1) lm-eval's key structural win is that raw model outputs are persisted (CachingLM sqlite, keyed only by request content — model.py:230-232) strictly upstream of all scoring (evaluator.py:596 vs :611/:635), so any metric/filter/judge change is re-scorable by re-running one identical command — Gym's rollouts.jsonl already plays this role, and a `gym eval verify` replay command would give Gym the same loop with a better artifact (readable JSONL vs prompt-hash sqlite) and without lm-eval's do_sample exclusion, which Gym avoids because it stores realized trajectories rather than caching by request key. (2) Because the cache is committed before scoring begins (model.py:264/:321), a scorer crash in lm-eval never destroys paid inference (if caching was on) — reinforcing that Gym must persist responses before/independently of verify. What NOT to copy: lm-eval has no judge-failure marker, no failed-subset resume with metric merge, judge crashes kill the whole run before anything is written (run.py:429-470), and framework judging remains open issues #2233/#1831 — so a verify_error field plus failure-class routing would leapfrog the incumbent rather than match it. One caveat added by verification: on current main the re-scoring use of --use_cache is no longer advertised (README now frames it purely as resume), so Gym should treat re-scoring as an explicit, documented contract rather than an emergent cache property.

## OpenCompass — `open-compass/opencompass`

**Architecture:** OpenCompass is explicitly two-phase: `-m infer` runs the policy model and writes per-task predictions to `outputs/<ts>/predictions/<model_abbr>/<dataset_abbr>.json`, while `-m eval` reads those predictions off disk and runs the evaluator/judge, writing scores to `outputs/<ts>/results/<model_abbr>/<dataset_abbr>.json` (opencompass/cli/main.py:357-443; path built by get_infer_output_path in opencompass/utils/abbr.py:40-48). Inference lives in inferencers (opencompass/openicl/icl_inferencer/icl_gen_inferencer.py); scoring/verification lives in evaluators (opencompass/evaluator/*.py, opencompass/openicl/icl_evaluator/) invoked by opencompass/tasks/openicl_eval.py, which never re-runs the policy model. This clean split is exactly what makes re-scoring possible without re-inference.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `first-class`**

OpenCompass separates generation from scoring at the CLI level. A prior run (`-m infer` or `-m all`) persists predictions under a timestamped work_dir. Re-scoring is `python run.py <config> -m eval -r <timestamp>`: `-m eval` runs only the eval phase (main.py:403-443) and `-r <timestamp>` points work_dir at the existing predictions folder (main.py:298-316; eval/viz mode without `-r` or a station raises ValueError at main.py:309-313). OpenICLEvalTask._load_predictions reads predictions/<model>/<dataset>.json from disk (openicl_eval.py:152-193) and _evaluate_predictions builds the evaluator freshly from the (edited) config each time (openicl_eval.py:254-314), so editing the metric, threshold, judge prompt or judge model in the config and re-running yields new rewards with zero policy inference. A related first-class route, `given_pred` (partitioners/base.py:34; subjective_eval.py:142-147), lets the eval task read predictions from an arbitrary external path — used to score against stored baseline answers.

```bash
python run.py <config> -m eval -r <timestamp>
```

Evidence:

- `opencompass/cli/main.py:81-99` — `-m eval` restricts to the eval phase (choices all/infer/eval/viz); `-r/--reuse` reuses prior outputs and skips finished tasks, taking `latest` (const) or an explicit timestamp e.g. 20230516_144254
- `opencompass/cli/main.py:298-316` — eval/viz mode requires `-r` (or --read-from-station); work_dir is set to the reused timestamp folder so predictions on disk are used
- `opencompass/tasks/openicl_eval.py:152-193` — eval task loads predictions from predictions/<model>/<dataset>.json off disk (including partial `_0`, `_1`... shards) — no model is invoked
- `opencompass/tasks/openicl_eval.py:254-314` — evaluator is rebuilt from the current (possibly edited) eval_cfg via ICL_EVALUATORS and scores the loaded predictions, so a changed metric/judge is applied to stored outputs
- `docs/en/user_guides/experimentation.md:55-56` — docs state `-m eval` runs only evaluation on existing model outputs in {WORKDIR}; `-r` reuses existing inference results and skips finished tasks
- `opencompass/tasks/subjective_eval.py:142-147` — `given_pred` config lets the judge eval task read predictions from an external path (stored baseline answers) rather than re-generating them

**Limitations:** Sharp edge: `-r` is designed to SKIP finished work, not to force re-scoring. Both the eval partitioner (partitioners/naive.py:68-70, partitioners/size.py:96-99) and OpenICLEvalTask.run (openicl_eval.py:96-97) skip any dataset whose results/<model>/<dataset>.json already exists. Because that path is keyed only on model+dataset abbr, not on the evaluator (utils/abbr.py:40-48), a changed metric produces the SAME path and gets skipped — so re-scoring an already-scored dataset requires manually `rm`-ing the stale results file (or reusing predictions into a fresh work_dir). Second, LLM-judge outputs are themselves cached: CascadeEvaluator reloads a cached _llm_judge_replicaN.json if present (evaluator/cascade_evaluator.py:204-227) and the judge GenInferencer resumes from a tmp jsonl, so a changed judge prompt can silently reuse stale judge verdicts unless those files are also removed. Only valid for evaluators that are pure functions of predictions+references+config; stateful/interactive evals are out of scope. Additional verified caveats the original missed: `-r <timestamp>` resolves inside cfg.work_dir, so the same `-w` work-dir must be passed as the original run; and `-r` with no argument picks the lexically-latest timestamp directory (main.py:299-307), which can silently reuse the wrong run if multiple experiments share a work_dir.

### PP04 — judge-only resume after scorer failures

**Support: `partial`**

**Failure recording:** There is no dedicated per-sample 'judge infrastructure failed' field distinct from a legitimate score. Three distinguishable outcomes exist. (1) Soft failure — judge returned text with no A/B verdict: _generic_llmjudge_postprocess maps it to 'unknown' (datasets/generic.py:63-70) and get_final_results buckets it as not_attempted_count, lowering attempted_ratio (datasets/generic.py:36-45,54,57); this is visible in aggregate but conflates 'judge declined', 'judge produced garbage', and mapped-error-string predictions. (2) Hard failure — judge API exhausts retries: OpenAISDK._generate raises RuntimeError (models/openai_api.py:854-855), which propagates and CRASHES the OpenICLEvalTask; in non-debug mode each task is its own subprocess (runners/local.py:234-248) so only that dataset dies while others finish, but in --debug the whole run dies. A crash is distinguishable by an absent results file, but it is destructive, not a marker. (3) Some paths hide the failure: OpenAISDK._generate returns a mapped error STRING as the prediction when the deployment configures status_code_mappings (openai_api.py:843; CORRECTED — this return is in OpenAISDK, not the legacy OpenAI class, and only fires when status_code_mappings is explicitly configured; the legacy OpenAI class raises RuntimeError after retries at openai_api.py:412-414), and OpenAISDKRollout returns dict(finish_reason='error') with the error message as output (openai_api.py:1105-1109) — both then get scored as if they were answers. Open feature request #2392 ('Judge Calibration Diagnostics') is real and open, but it concerns judge calibration/ranking validity (CJE, oracle labels), not the infra-failure-recording gap; no upstream issue tracking the recording gap itself was verified.

**In-run retry:** Judge model calls retry `self.retry` times (default 2, models/base_api.py:46,54). On exhaustion OpenAISDK raises RuntimeError (models/openai_api.py:732 `while num_retries < self.retry`, 854-855). The legacy OpenAI._generate handles rate_limit_exceeded by sleeping 10s and `continue`-ing WITHOUT reaching the `max_num_retries += 1` at loop end (openai_api.py:389-392, increment at ~410) — effectively unbounded retry/stall on a rate-limited judge; insufficient_quota drops the key and retries (openai_api.py:393-396). A separate runner-level `--retry` (default 2, cli/main.py:136-141) resubmits failed slurm/dlc jobs and is unrelated to per-call judge retries.

**Resume:** Genuine judge-only resume with no policy re-inference works via the two-phase split plus the judge inferencer's checkpoint. Predictions are already on disk, so `-m eval -r <ts>` never touches the policy model. The GenericLLMEvaluator drives the judge through a GenInferencer (evaluator/generic_llm_evaluator.py:80) whose inference() restores completed judge calls from tmp_<name>.jsonl (icl_gen_inferencer.py:118-130; restore_from_jsonl at icl_base_inferencer.py:130-150) and starts the dataloader at prompt_list[index:] (icl_gen_inferencer.py:130). So a judge run that crashed midway resumes at the first un-judged sample on re-run — only the failed subset gets fresh judge calls. Merge is automatic: the results JSON is written only once the eval task completes (openicl_eval.py:424-432) and metrics are recomputed over the full union of resumed+new judge outputs (datasets/generic.py:96-101), so a resumed run's metrics are identical to a clean run and `-m viz` summarizes them the same way. Caveat: resume granularity is positional index in the tmp jsonl, not a per-sample failure marker — a sample whose judge call 'succeeded' with a bad/error string is considered done and is NOT retried; only samples never reached are re-judged.

```bash
python run.py <config> -m eval -r <timestamp>
```

Evidence:

- `opencompass/models/openai_api.py:854-855` — judge (OpenAISDK) raises RuntimeError after exhausting retries, crashing the eval task rather than recording reward=0
- `opencompass/models/base_api.py:46` — judge model retry defaults to 2 (retry: int = 2; stored as self.retry at line 54)
- `opencompass/models/openai_api.py:389-392` — legacy OpenAI path retries rate_limit_exceeded indefinitely (sleeps 10s, continues past the max_num_retries increment) — a rate-limited judge stalls rather than fails
- `opencompass/models/openai_api.py:843` — CORRECTED: it is OpenAISDK._generate (not legacy OpenAI.generate) that returns a mapped error string as the prediction, and only when status_code_mappings is configured; the string is then scored as an answer
- `opencompass/models/openai_api.py:1105-1109` — OpenAISDKRollout maps configured status codes to dict(finish_reason='error') with the error message as output, which is then scored as an answer
- `opencompass/openicl/icl_inferencer/icl_gen_inferencer.py:118-130` — judge GenInferencer restores completed calls from tmp_<name>.jsonl and resumes the dataloader at the first un-judged index (prompt_list[index:])
- `opencompass/openicl/icl_inferencer/icl_base_inferencer.py:130-150` — restore_from_jsonl repopulates results_dict/dumped_indices so resumed judge samples are not re-called (corrupt tmp files are moved to .bak and resume restarts from scratch)
- `opencompass/tasks/openicl_eval.py:96-97` — eval task skips a dataset whose results file already exists; a crashed (results-less) judge run therefore re-runs and resumes on `-m eval -r`
- `opencompass/datasets/generic.py:36-45` — unparseable/'unknown' judge verdicts are counted as not_attempted_count (surfaced via attempted_ratio), giving an aggregate signal that conflates several failure/soft-failure modes
- `opencompass/evaluator/cascade_evaluator.py:204-227` — cascade LLM-judge results are cached to disk and reloaded (with a strict length check that raises on mismatch) so only uncached failed samples trigger new judge inference
- `opencompass/runners/local.py:234-248` — in non-debug mode each task runs as a subprocess whose non-zero return is logged while other tasks continue, so one dataset's judge crash does not kill the whole run
- `https://github.com/open-compass/opencompass/issues/2392` — open feature request 'Add Judge Calibration Diagnostics for LLM-as-Judge Evaluation' — about judge calibration/ranking validity, adjacent to but NOT specifically about distinguishing judge infra failures from low scores

**Limitations:** No standalone 're-run only failed judge samples' command and no per-sample failure marker: resume is a side effect of (a) the results-file-existence skip and (b) positional tmp-jsonl checkpointing, not a first-class judge-retry API. A hard judge failure crashes the eval task (destroying in-process state) rather than persisting a recoverable failure row, and in --debug single-process mode it takes the whole run down. Samples where the judge returned a bad/error STRING are treated as completed and are never retried by resume. There is no schema field separating 'judge infra error' from 'task scored low', so metrics can silently absorb judge failures as not_attempted or as scored error strings. Corrections/additions from verification: the mapped-error-string swallowing is an OpenAISDK behavior gated on the optional status_code_mappings config (the legacy OpenAI class crashes with RuntimeError instead, openai_api.py:412-414); issue #2392 evidences upstream concern about judge score reliability generally, not this recording gap specifically; and restore_from_jsonl silently discards a corrupt tmp checkpoint (moved to .bak, icl_base_inferencer.py:142-147), so a resume after tmp-file corruption re-judges ALL samples.

**Lessons for Gym:** OpenCompass ships exactly the PP02 primitive Gym lacks: a clean generation/scoring phase split where `-m eval -r <timestamp>` re-scores predictions already on disk with an edited evaluator/judge and zero policy inference — Gym should ship an analogous `gym eval verify` that replays rollouts.jsonl through `/verify`, since it already persists the expensive `response`. For PP04, OpenCompass gets judge-only resume nearly for free by checkpointing every judge call to a tmp jsonl (save_every, default 1) and resuming from the first un-judged sample; Gym could persist inference before the verify call and checkpoint judge outputs so a resume re-judges only failed samples and merges into identical metrics. The cautionary lessons are equally useful: OpenCompass's reuse SKIPS finished work keyed on model+dataset only, so re-scoring already-scored data needs a manual file delete, hard judge failures crash the task, and error strings from a judge deployment with status_code_mappings are silently scored as answers with no failure marker distinct from a low score — Gym should instead add an explicit verify_error/failure-class field and a force-rescore flag so failures are recorded distinguishably and re-scoring is intentional rather than a delete-file side effect.

## SWE-bench evaluation harness — `SWE-bench/SWE-bench`

**Architecture:** Policy inference never happens inside the evaluation harness: the harness consumes a finished predictions .json/.jsonl of {instance_id, model_name_or_path, model_patch} (loaded in swebench/harness/utils.py:41-77; optional generation scripts live in the separate swebench/inference/ package). Scoring is a two-stage per-instance pipeline in swebench/harness/run_evaluation.py:71-273 — (1) a docker container applies the patch and runs the repo test suite, writing raw evidence to logs/run_evaluation/<run_id>/<model>/<instance_id>/test_output.txt (constants at swebench/harness/constants/__init__.py:19,74-76), then (2) pure-Python grading parses that log into report.json (swebench/harness/grading.py:39-91, 235-295). Aggregate metrics are recomputed from the on-disk report.json tree by make_run_report (swebench/harness/reporting.py:17-160), never frozen at collection time.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `first-class`**

Generation and scoring are separated by construction: the scorer's only input is a predictions file, so any verifier change plus a re-run of run_evaluation re-pays zero model inference. There are two tiers of re-scoring. Tier 1 — if the verifier change is in log parsing or grading logic (swebench/harness/log_parsers/, grading.py), `--rewrite_reports True` re-grades the stored test_output.txt for every instance without starting any container: run_instance takes an early path that calls get_eval_report on the saved log and rewrites report.json (run_evaluation.py:101-117), the dataset is pre-filtered to instances whose test_output.txt exists (run_evaluation.py:412-434), and env-image builds are skipped (run_evaluation.py:542); make_run_report then recomputes aggregates from the rewritten report.json files (reporting.py:51-87,127-160). Tier 2 — if the change requires re-executing tests (eval scripts, FAIL_TO_PASS/PASS_TO_PASS lists, install specs in test_spec), you re-run the same predictions under a new run_id: containers re-run (the cheap-ish deterministic scorer), but the expensive half (the model) is untouched because it was never in the loop. The user must have saved: the predictions file and, for tier 1, the logs/run_evaluation/<run_id>/... tree (patch.diff, test_output.txt); the SWE-bench/experiments repo publicly archives exactly these artifacts so third parties can re-grade published runs.

```bash
python -m swebench.harness.run_evaluation --dataset_name SWE-bench/SWE-bench_Lite --predictions_path preds.jsonl --run_id my_run --rewrite_reports True
```

Evidence:

- `swebench/harness/run_evaluation.py:101-117` — rewrite_reports early path re-grades stored test_output.txt via get_eval_report and rewrites report.json without building or starting a container
- `swebench/harness/run_evaluation.py:412-434` — with rewrite_reports, get_dataset_from_preds returns only instances whose test_output.txt exists on disk under logs/run_evaluation/<run_id>/<model>/<instance_id>/
- `swebench/harness/run_evaluation.py:542-565 and 663-668` — env-image builds are skipped when rewrite_reports is set (line 542, `if namespace is None and not rewrite_reports`); the flag is a documented argparse option ('Doesn't run new instances, only writes reports for instances with existing test outputs')
- `swebench/harness/grading.py:39-91` — get_logs_eval re-parses the raw test log with the current MAP_REPO_TO_PARSER log parser (line 53) — the exact component a verifier iteration would change
- `swebench/harness/utils.py:41-77` — predictions (the model's expensive output) enter the harness only as a static .json/.jsonl file (or 'gold' to pull dataset gold patches); the harness contains no policy-inference call
- `swebench/harness/reporting.py:51-87` — make_run_report recomputes aggregate metrics by re-reading every per-instance report.json, so re-graded instances flow into the final report automatically

**Limitations:** rewrite_reports covers only the log-parse/grading half of the verifier; changes to eval scripts or F2P/P2P test lists require re-running containers (still no inference). It requires test_output.txt for the targeted instances — the ValueError at run_evaluation.py:102-104 exists but in practice never fires from main() because the pre-filter (412-434) SILENTLY DROPS instances whose test_output.txt is missing; they simply land in error_ids with no warning. It rewrites report.json in place — old grades are destroyed, no side-by-side comparison (and the aggregate <model>.<run_id>.json is overwritten too). The local path still constructs a docker client unconditionally (run_evaluation.py:535, plus run_instances:303), so pure re-grading needs a running docker daemon. Timed-out runs get a plain 'Timeout error: N seconds exceeded.' string appended (run_evaluation.py:214-215) which does NOT match the TESTS_TIMEOUT bad-code marker '>>>>> Tests Timed Out' checked by get_logs_eval (grading.py:61-73, constants/__init__.py:89), so re-grading silently converts what was originally a hard error (no report.json) into a legitimate-looking resolved=false report from a truncated log. The flag is undocumented in docs/ (verified: zero mentions in docs/, CHANGELOG.md, README.md) and relatively recent: it landed in commit d83e100 'SWE-bench Multimodal Dev (#285)', committed 2025-01-13 (verified via git log -S rewrite_reports; repo clone is full, not shallow).

### PP04 — judge-only resume after scorer failures

**Support: `first-class`**

**Failure recording:** A scorer failure (docker build error, patch-apply failure, test timeout, daemon error — there is no LLM judge; the scorer is deterministic test execution) is recorded as the ABSENCE of report.json plus a full traceback in the per-instance run_instance.log: run_instance catches EvaluationError/BuildImageError and generic Exception (run_evaluation.py:253-263) and returns {completed: False} from the finally block (270-273) without writing report.json. A legitimate low score writes report.json with resolved=false. Post-hoc the two are cleanly distinguishable: make_run_report puts no-report instances into error_ids (reporting.py:85-87) and graded-but-failed instances into unresolved_ids (reporting.py:77-81); empty or corrupt report.json also goes to error_ids (reporting.py:70-84), and the final report JSON lists error_ids, unresolved_ids, resolved_ids separately (reporting.py:127-143). What is NOT structured is the failure CAUSE — only tracebacks in logs; open PR #594 ('fix: add infrastructure failure classification to evaluation pipeline', opened June 2 2026, claiming an audit found 62.5% of sampled unsolved-by-all instances fail due to environment bugs) and issue #472 (asks docs to define 'Instances with errors') show this gap is felt by users.

**In-run retry:** Local path: no automatic per-instance retry at all — each failure is caught once, logged, and counted in the tqdm 'error' stat (run_evaluation.py:253-263, 352-368); run_threadpool additionally catches and collects any escaping exception so one bad instance never kills the batch (swebench/harness/utils.py:80-108). Modal path: tenacity retry with 7 attempts and exponential backoff, but only around sandbox creation for network flakiness (swebench/harness/modal_eval/run_evaluation_modal.py:68-71); test execution itself is not retried, and starmap uses return_exceptions=True so failures are printed and skipped (run_evaluation_modal.py:437-443).

**Resume:** Re-run the exact same command with the same --run_id. get_dataset_from_preds scans the run's log tree and excludes every instance whose report.json exists ('N instances already run, skipping...', run_evaluation.py:436-456), so only errored instances (which have no report.json) are re-scored; run_instance also short-circuits per-instance on report_path.exists() (run_evaluation.py:118-123). No inference occurs because the harness never does inference — the predictions file is reused as-is. Merging is automatic and exact: make_run_report ignores in-memory state and recomputes all counts/id-lists by re-reading every per-instance report.json across the full dataset (reporting.py:51-87), so after a resume the final <model>.<run_id>.json is byte-equivalent to what a clean run would have produced. This is designed-for behavior (exclude_completed defaults to True, run_evaluation.py:381) though the docs never describe it.

```bash
python -m swebench.harness.run_evaluation --dataset_name SWE-bench/SWE-bench_Lite --predictions_path preds.jsonl --run_id my_run   # identical rerun; completed instances skip, only errored ones re-score
```

Evidence:

- `swebench/harness/run_evaluation.py:253-273` — scorer failures are caught (EvaluationError/BuildImageError/Exception), logged to run_instance.log, and return completed=False without writing report.json — the run continues
- `swebench/harness/run_evaluation.py:436-456` — resume: with the same run_id, instances with an existing report.json are skipped ('already run, skipping...'), so exactly the failed subset is re-scored
- `swebench/harness/run_evaluation.py:118-123` — per-instance idempotency: run_instance returns the cached report if report.json already exists
- `swebench/harness/reporting.py:68-87 and 127-143` — error_ids (no/empty/corrupt report.json) vs unresolved_ids (graded, resolved=false) are recorded as separate lists in the final run report — infra failure and low score are distinguishable post-hoc
- `swebench/harness/utils.py:80-108` — run_threadpool catches per-instance exceptions and collects them into a failed list rather than crashing the batch
- `swebench/harness/modal_eval/run_evaluation_modal.py:68-71` — the only automatic retry in the codebase: tenacity 7-attempt exponential backoff (min 4s, max 10s) on Modal sandbox creation
- `https://github.com/SWE-bench/SWE-bench/pull/594` — open PR 'fix: add infrastructure failure classification to evaluation pipeline' (opened 2026-06-02) proposes a structured INFRA_FAILURE classification, evidence the unstructured traceback-only failure cause is a recognized gap
- `https://github.com/SWE-bench/SWE-bench/issues/294` — gold-patch instability (14 unresolved on first run, 15 on second, on SWE-bench_Verified) shows resume covers hard errors only — flaky low scores write report.json and are never retried

**Limitations:** Resume keys purely on report.json existence, so it recovers hard errors only: a flaky test run that produced resolved=false (issue #294) counts as completed and is never re-scored without manually deleting its report.json or using --rewrite_reports/a new run_id. The Modal path's resume is broken relative to local for a subset of failures: when run_instance_modal catches EvaluationError/Exception it returns a TestOutput with errored=True and report_json_str='' (run_evaluation_modal.py:364-391), and run_instances_modal then writes the local log_dir including an EMPTY report.json (run_evaluation_modal.py:454-460 opens report.json with 'w' before json.loads fails); since run_instances_modal skips any instance whose log_dir merely exists (run_evaluation_modal.py:417-423) and get_dataset_from_preds treats the empty report.json as completed, such instances are silently never re-run and stay in error_ids until the user deletes their log dirs. (Raw starmap exceptions, by contrast, write no local log_dir at all — run_evaluation_modal.py:440-443 — so those DO re-run on resume.) Failure cause is unstructured (traceback text in run_instance.log only); there is no in-run retry on the local path even for transient docker daemon errors. None of the resume behavior is documented in docs/ — it is discovered from console output or source.

**Lessons for Gym:** Two directly copyable ideas. For PP02: SWE-bench splits the verifier itself into an expensive evidence-producing stage (test_output.txt) and a cheap grading stage (report.json), persists both per-sample keyed by run_id/instance_id, and always recomputes aggregates from the artifact tree — which makes `--rewrite_reports` (re-grade stored evidence with changed grading code, run_evaluation.py:101-117) a ~17-line feature; Gym's `gym eval verify` should similarly treat rollouts.jsonl + materialized inputs as the evidence tier and make aggregation always artifact-derived. For PP04: represent scorer failure as the ABSENCE of the per-sample verdict artifact (never reward=0), so resume is just 'skip if verdict exists' (run_evaluation.py:436-456) and error_ids vs unresolved_ids stay distinguishable in reports (reporting.py:68-87) — Gym's dispatcher could adopt the same invariant by persisting the response and routing verify failures to a sidecar with no reward field. Cautionary tales to avoid: leaving both features undocumented; the Modal-path bug where writing an empty verdict file on failure permanently blocks resume; and the timeout-marker mismatch (run_evaluation.py:215 vs constants TESTS_TIMEOUT) where re-grading a truncated log silently converts an infra error into a legitimate-looking low score — keep failure markers and grading checks in one shared constant.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `workaround-possible`**

### PP04 — judge-only resume after scorer failures

**Support: `absent`**

**Failure recording:** Two layers, both distinguishable post-hoc — notably better than a bare low score. (1) Client-observed failures get a typed error enum (TaskError: START_FAILED, INTERACT_FAILED, AGENT_FAILED, NETWORK_ERROR, NOT_AVAILABLE — src/client/task.py:10-15) and are written to a separate error.jsonl sidecar, never runs.jsonl, so they are excluded from metrics (src/assigner.py:368-378); the row preserves the partial trajectory because run_sample returns output=latest_result on agent/interact failures (src/client/task.py:92-94, :117-119). (2) Worker-side scorer/environment exceptions do NOT crash anything: they are caught and returned as a completed sample with status='task error' (SampleStatus.TASK_ERROR, src/typings/status.py:12; dbbench task.py:242-245), recorded inside runs.jsonl — os_interaction even stores the full traceback in result ({'result': False, 'error': traceback}, src/server/tasks/os_interaction/task.py:388-392) — and surfaced as a separate 'task error' rate in overall.json's validation statistics (src/client/task.py:128-135).

**In-run retry:** Model-API calls: 3 attempts with incremental sleep (time.sleep(attempt+2)) inside HTTPAgent.inference (src/client/agents/http_agent.py:189-215). The FC-era AgentRL eval client also retries 3x with 1s delay (AgentRL examples/eval/server_agent.py, chat_completion max_retries=3 at ~L72, asyncio.sleep(1) at ~L86; verified in local clone @ 6a73409). Sample-level: the assigner's finish_callback ALWAYS requeues NOT_AVAILABLE samples (src/assigner.py:332-343), and with --auto-retry (argparse at src/assigner.py:414-415) requeues any errored sample (src/assigner.py:345-350) — uncapped, and each retry is a full new session including all policy inference. There is no retry that targets only the scoring step, because scoring is not a separable call.

**Resume:** Scoring-only resume is architecturally impossible and absent: the reward/verdict is produced inside the live env session (legacy: TaskSampleExecutionResult at answer-commit time, dbbench task.py:209-225; FC: reward returned by /interact on finish, AgentRL server_agent.py:144-145) and env state is torn down at session end, so there is nothing to re-judge against. What exists is first-class WHOLE-SAMPLE resume: re-running the assigner against an existing output dir reloads runs.jsonl, skips finished samples, and re-dispatches only missing/errored ones — through full inference (src/assigner.py:70-117; documented at docs/Entrance_en.md:62-63). Merge semantics are clean: resumed rows append to runs.jsonl and overall.json is computed over the union exactly as in a clean run (assigner.py:301-327). The gap: TASK_ERROR rows (scorer/env failures) live in runs.jsonl with error=None and are treated as finished — resume never retries them; recovering them requires hand-deleting lines from runs.jsonl and re-paying full inference. The FC client resumes the same way: -f skips rows where result != 'error' (server_agent.py completed_runs ~L267-271, skip loop ~L309-314) and re-runs errored samples end-to-end.


AgentBench is also a cautionary example for PP02: it fuses per-sample verification into the live env session even more deeply than Gym does, making stored trajectories permanently un-re-scorable — reinforcing the recommendation that servers declare replayability (supports_offline_verify) rather than letting statefulness silently foreclose re-scoring.

## Inspect AI — `UKGovernmentBEIS/inspect_ai`

**Architecture:** Inference (solver/agent loop) and scoring both run inside the per-sample task runner (src/inspect_ai/_eval/task/run.py: solver plan first, then scorers at run.py:1496-1570), but they are decoupled through the eval log artifact: every EvalSample persists input, target, full message history, model output, events, and store (src/inspect_ai/log/_log.py:378-458). Because the log is a complete record, scoring is also a standalone phase: src/inspect_ai/_eval/score.py (score_async) reconstructs a TaskState purely from the stored sample (score.py:408-423, completed=True, no generation) and re-runs scorers/metrics, exposed via the `inspect score` CLI (src/inspect_ai/_cli/score.py:41-154). Metrics are always recomputed from sample scores (eval_results, score.py:357-366), never frozen separately.

### PP02 — re-score stored rollouts with a changed verifier

**Support: `first-class`**

Every eval run writes a .eval log whose samples contain the full trajectory (messages, output, events, store, target). `inspect score <log>` reads that log, resolves scorers either from the log header (log.eval.scorers), from log.results, or from a user override (--scorer name|file.py@name with -S args), rebuilds a completed TaskState per sample from the stored fields, and runs the scorer(s) with zero policy-model generation. Metrics are then recomputed from the new scores via eval_results and merged into the log with --action append (new scores added alongside old, name-deduped via unique_scorer_name) or overwrite (results replaced); score events are spliced into each sample's event tree. Since 0.3.242 (June 2026) you can also swap the grader model at re-score time (--model / --model-role), so LLM-judge hyperparameters (model, prompt via -S, metrics via --metric) can all change without re-inference. The inverse workflow is also supported: run with --no-score, score later.

```bash
inspect eval task.py --no-score   # optional deferred scoring
inspect score ./logs/<logfile>.eval --scorer custom_scorers.py@classify -S threshold=0.5 --action overwrite --output-file rescored.eval   # or --model/--model-role to swap the judge; Python: score(log, scorers, action=...)
```

### PP04 — judge-only resume after scorer failures

**Support: `partial`**

**Failure recording:** A scorer/judge exception is caught per-sample (run.py:1604-1614, handle_error inside the scoring try-block) and recorded as a structured sample-level error: EvalSample.error is an EvalError with message/traceback (log/_log.py:494-498, plus error_retries history at 497-498), an ErrorEvent lands in the sample transcript (run.py:1602), and the sample has no score for that scorer — categorically distinct from a legitimate low score, and queryable post-hoc (read_eval_log_sample_summaries surfaces s.error, run.py:1929-1931). Crucially the errored sample is still written to the log with its full messages/output (shielded logging at run.py:1669-1693), so the paid inference is never lost. Soft judge failures are weaker: built-in model-graded scorers coerce an unparseable verdict to INCORRECT (scorer/_model.py:228-240) — detectable post-hoc via the 'Grade not found in model output' explanation and the full grading exchange in score metadata, but it silently pollutes the headline metric, same half-measure as NeMo Gym's verdict_label:null.

**In-run retry:** Two layers. (1) Model-API layer: judge calls inside scorers go through get_model().generate() (scorer/_model.py:208), which retries transient errors (per-provider should_retry classification) with tenacity exponential backoff + jitter, initial 3s, capped at 30-min waits, and by default retries FOREVER (stop_never) unless max_retries/timeout set (model/_retry.py:61-66 wait, 94-107 stop; ref 37-105 verified); UNVERIFIED: non-transient errors (401/400) fail immediately (consistent with the transient-only should_retry design but the per-provider classifiers were not individually checked). (2) Sample layer: an exhausted/hard scorer failure becomes a sample error subject to fail_on_error (bool/proportion/count threshold, default True = fail immediately, docs/handling-errors.qmd:52); --retry-on-error=N re-runs the WHOLE sample in-run, inference included (recursion with retry_on_error - 1 at run.py:1734-1775); --score-on-error scores the partial state instead of failing (run.py:1505-1512, docs/handling-errors.qmd:92-119). Scoring also gets its own time limit of half the sample timeout (run.py:1482-1488).

**Resume:** `inspect eval-retry <log>` re-runs only non-clean samples: eval_log_sample_source reuses any prior sample with error=None (and invalidation=None) verbatim — its scores are copied straight into the new run's results with no re-inference and no re-scoring (run.py:629-672, 2025-2027) — and metrics are recomputed over reused+rerun samples together, so the final log is indistinguishable from a clean run. For the errored (judge-failed) samples themselves there are two cases. With checkpointing enabled (--checkpoint, eval.py:405, feature added in 0.3.241, 22 June 2026): a final 'agent_complete' checkpoint fires at agent exit (checkpointer_impl.py:160-178), so a sample that died during scoring resumes as attempt='resume_for_scoring' (run.py:1944-1957) — checkpoint-aware agents like react() restore tracked state and return immediately, skipping the entire agent loop so ONLY scoring re-runs (_react.py:208-213) — this is literally judge-only resume by design. Without checkpointing: the errored sample is re-run from scratch, re-paying its policy inference (PreviousError path, run.py:1959-1978). Alternative zero-inference path: `inspect score` on the errored log re-scores from stored outputs, but hits ALL samples, not just failed ones.

```bash
inspect eval-retry ./logs/<logfile>.eval   # re-runs only errored samples; add --checkpoint on the ORIGINAL run to make this scoring-only (resume_for_scoring). Zero-inference alternative for all samples: inspect score ./logs/<logfile>.eval --action overwrite
```

**Lessons for Gym:** First, make the rollout artifact self-sufficient and make re-scoring a command, not a script: Inspect's EvalSample keeps everything the verifier needs (messages, output, target, store) in one row, and `inspect score` (with --scorer/-S overrides, append-vs-overwrite score actions, and metric recomputation from the merged log) is precisely the `gym eval verify` proposed in PP02 §5 — including the --no-score flag as the inverse workflow and a declared-capability caveat Gym should copy as supports_offline_verify (Inspect's silent sandbox-scorer failure shows what happens without it). Second, for PP04, Inspect's two-part design maps directly onto Gym's fix hooks: (a) always persist completed inference WITH a structured sample-level error when scoring fails (EvalSample.error, richer than the proposed _ng_failure_class='verify' sidecar row), so retry can even exist; (b) the 'resume_for_scoring' checkpoint attempt — agent finished cleanly, scoring crashed, retry restores state and skips the agent loop — is a shipping blueprint for Gym's 'inference done, judge pending' resume state, and eval-retry's verbatim reuse of clean samples plus fresh metric computation over the merged set is exactly the 'metrics identical to a clean run' acceptance criterion; note it is opt-in, dev-labelled, and requires checkpoint-aware scaffolds, so Gym could differentiate by making judge-pending resume the default rather than opt-in.
