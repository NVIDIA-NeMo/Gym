This page documents the automated checks that run on every pull request and on every push to `main`. It covers what runs, when it runs, and what the checks actually enforce.

---

## PR Checks (pre-merge)

These checks run on every PR and must all pass before merging. Each is defined in a separate workflow under `.github/workflows/`.

### 1. Code Linting

**Workflow:** `.github/workflows/code-linting.yml`

Runs `pre-commit run --all-files --show-diff-on-failure` across the full repository. The following hooks are enforced:

| Hook | What it checks |
|------|----------------|
| `end-of-file-fixer` | Python files end with a newline |
| `trailing-whitespace` | No trailing whitespace in Python files |
| `ruff` (lint) | Python linting with auto-fix |
| `ruff` (imports) | Import order (`isort`-style) with auto-fix |
| `ruff-format` | Code formatting |
| `no-underscore-md` | No underscores in Markdown filenames |
| `add-verified-flag` | New resources server YAML configs get `verified: false` injected automatically |
| `update-readme-table` | Root `README.md` environment table is kept in sync |

The `add-verified-flag` hook means any new resources server YAML will be committed with `verified: false`. This field must be manually flipped to `true` after the benchmark is baselined and reviewed (see [post-merge](#post-merge-steps)).

### 2. Copyright Check

**Workflow:** `.github/workflows/copyright-check.yml`

Every new file must include the Apache 2.0 copyright header. See the [Development Setup](./development-setup) page for the exact header text.

### 3. DCO Sign-off

All commits must carry a `Signed-off-by` line (`git commit -s`). See [Development Setup](./development-setup) for setup instructions.

### 4. Fern Docs Check

**Workflow:** `.github/workflows/fern-docs-ci.yml`

Runs `npm run check` inside `fern/` to validate the documentation configuration. Triggered when the FW-CI mirror bot pushes to a `pull-request/[0-9]+` branch.

### 5. Smart Unit Test Selection

**Workflow:** `.github/workflows/unit-tests.yml`

The CI classifies every PR's changed files into one of three buckets, and runs the appropriate test scope:

| Changed files | Test scope |
|--------------|------------|
| Only `**.md`, `docs/**`, `fern/**`, `LICENSE`, `benchmarks/**` | **Skip** — no tests run |
| Only `resources_servers/**`, `responses_api_agents/**`, `responses_api_models/**` | **Server-only** — test only the changed servers |
| Anything else (core library, CI, scripts, etc.) | **Full suite** — all tests run |

The priority is `other > server > doc`: a PR that touches both a server file and a core file runs the full suite.

**Core library tests** (when full suite runs):
```
pytest tests/unit_tests/ -m "not sandbox"   # main pass
pytest tests/unit_tests/ -m sandbox          # sandbox pass
```

**Server-only tests** (when only server files changed):
- Extracts the changed server names from the diff
- Runs `gym env test --resources-server <name>` for each
- `delete_venvs_after_each_test=true` for isolation

### 6. Server Suite (sharded)

When the full suite runs, the server tests are split across **8 parallel shards**:

```
ng_test_all +fail_on_total_and_test_mismatch=true +delete_venvs_after_each_test=true +num_shards=8 +shard_index=<0-7>
```

The key flag is `fail_on_total_and_test_mismatch=true`: **every resources server module must have at least one test**. A server with no `tests/test_app.py` fails the shard that picks it up.

A **server-suite-result** aggregation job waits for all 8 shards and fails the PR if any shard fails.

### 7. Data Validation (inside `gym env test`)

When `gym env test` runs (either the server-only path or as part of the full sharded suite), it calls `_validate_data_single()` on each resources server. This enforces:

| File | Requirement |
|------|-------------|
| `data/example.jsonl` | Must exist with **exactly 5 rows** |
| `data/example_metrics.json` | Must exist; `"Number of examples"` must equal `5` |
| `data/example_rollouts.jsonl` | Must exist with **exactly 5 rollouts** |
| `data/` directory | Must contain **no `*conflict*` files** (git merge artifacts) |

Missing files produce an error message with the exact command to generate them.

### 8. Config Assertions

These are checked when the server is loaded during `gym env test`:

- `domain` is required in every resources server YAML config
- Train and validation datasets must include a `license` field

## Full Test Suite (post-merge / main branch)

**Workflow:** `.github/workflows/full-test-suite.yml`

Runs on every push to `main` and on-demand via `workflow_dispatch`. No change detection — always runs everything.

### Jobs

**`full-test-suite`** — Core library tests (identical markers to the PR suite):
```
pytest tests/unit_tests/ -m "not sandbox"
pytest tests/unit_tests/ -m sandbox
```

**`server-suite`** — Same 8-shard server test run as in PR CI, with `fail_on_total_and_test_mismatch=true`.

**`test-wheel-install`** — End-to-end integration test:
1. Builds the package wheel and serves it as a local PyPI index
2. Installs the wheel in a fresh venv
3. Starts a mock inference endpoint
4. Runs: `ng_help`, `ng_dump_config`, `ng_init_resources_server`
5. Runs `ng_run` + `ng_collect_rollouts` against a built-in server (`arc_agi`)
6. Runs `ng_run` + `ng_collect_rollouts` against a local server (`hello_world`)

**`notify-failure`** — Posts a Slack notification if any of the above jobs fail.

## Post-Merge Steps

These are not automated CI gates but are required before a contributed environment is considered production-ready.

**`verified: false` → `verified: true`**: After merging, a maintainer must baseline the benchmark (reward profiling, training validation) and flip `verified: true` in the YAML. Until then, the server is present in the repo but not surfaced as a verified environment.

See [Adding a Benchmark](./environments/adding-a-benchmark) for the full baselining requirements.
