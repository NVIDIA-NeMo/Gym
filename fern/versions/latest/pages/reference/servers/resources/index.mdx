The Resources server hosts the **environment side of the rollout**: the tools the agent can call, any sandbox or external state the environment maintains, and the `verify()` method that scores the final output. Per-rollout state is keyed by session cookie so concurrent rollouts don't bleed into each other.

## Endpoints

`SimpleResourcesServer` (the canonical base class) registers these routes via `setup_webserver` (`nemo_gym/base_resources_server.py:60-69`):

| Method | Path | Body | Returns |
|---|---|---|---|
| `POST` | `/seed_session` | `BaseSeedSessionRequest` (extensible per env) | `BaseSeedSessionResponse` |
| `POST` | `/verify` | `BaseVerifyRequest` (request + response) | `BaseVerifyResponse` (`reward: float` plus per-env fields) |
| `POST` | `/aggregate_metrics` | `AggregateMetricsRequest` (rollouts JSONL) | `AggregateMetrics` |
| `POST` | `/<tool_name>` | per-tool request body | per-tool response body |

Subclasses can register additional tool endpoints in their own `setup_webserver` override.

## Base class hierarchy

```
BaseServer
└── BaseResourcesServer (carries Pydantic config)
    └── SimpleResourcesServer (FastAPI + session middleware + AggregateMetricsMixin)
```

`SimpleResourcesServer` is the entry point for environment authors. It implements `seed_session` as a no-op default and leaves `verify` abstract — the only required method to implement.

## Request and response shapes

The shapes are defined in `nemo_gym/base_resources_server.py:37-54`:

```python
class BaseRunRequest(BaseModel):
    responses_create_params: NeMoGymResponseCreateParamsNonStreaming

class BaseVerifyRequest(BaseRunRequest):
    response: NeMoGymResponse

class BaseVerifyResponse(BaseVerifyRequest):
    reward: float

class BaseSeedSessionRequest(BaseModel):
    pass

class BaseSeedSessionResponse(BaseModel):
    pass
```

`BaseVerifyResponse` is the only one with a hard-required field beyond what subclasses add. The `reward: float` contract has no enforced range — `[0, 1]` is convention; partial-credit envs return values in between, and a few envs return negative values for penalty-shaped rewards. Subclasses extend the response with per-environment fields (extracted answers, judge rationales, per-test breakdowns).

## Implementing a verifier

The minimum a `SimpleResourcesServer` subclass must provide:

```python
from nemo_gym.base_resources_server import (
    BaseVerifyRequest,
    BaseVerifyResponse,
    SimpleResourcesServer,
)

class MyEnvServer(SimpleResourcesServer):
    async def verify(self, body: BaseVerifyRequest) -> BaseVerifyResponse:
        model_text = body.response.output_text
        expected = body.responses_create_params.input  # or a custom field
        reward = 1.0 if model_text.strip() == expected else 0.0
        return BaseVerifyResponse(**body.model_dump(), reward=reward)
```

For tool endpoints, override `setup_webserver` and register the routes:

```python
def setup_webserver(self):
    app = super().setup_webserver()
    app.post("/calculate")(self.calculate)
    return app

async def calculate(self, request: Request, body: CalculateRequest) -> CalculateResponse:
    ...
```

## Session middleware

`SimpleResourcesServer.setup_webserver` calls `self.setup_session_middleware(app)` (`nemo_gym/base_resources_server.py:63`). The middleware assigns each rollout a unique `session_id` and propagates it via cookies. Stateful environments key per-rollout state on `request.session[SESSION_ID_KEY]`.

For the cookie-propagation contract that custom agent harnesses must implement to keep session state alive, refer to [Stateful Environment](/latest/build-environments/supported-patterns/stateful#cookie-propagation-contract-required-for-stateful-agents).

## Reward contract

| Behavior | Source |
|---|---|
| `reward: float` is required on every verify response. | `nemo_gym/base_resources_server.py:46` |
| No range is enforced; `[0, 1]` is convention. Partial credit and penalties are allowed. | Per shipped envs (e.g. `equivalence_llm_judge` returns 0.5 for partial matches). |
| Errors must not crash the rollout; `verify()` should catch its own exceptions and return `reward=0.0` with an error field. | Per `resources_servers/AGENTS.md` invariants. |
| Aggregate metrics (mean/max/min/median/std per numeric field) are computed by `RewardProfiler`. | `nemo_gym/reward_profile.py:54-74`. |

For the full reward contract — what happens on errors, retries, multiple rollouts, pass@k aggregation — refer to [Reward Semantics](/latest/about/concepts/reward-semantics). For the diagnostic playbook when reward is wrong, refer to [Reward Debugging](/latest/about/concepts/reward-debugging).

## Example resources servers

**[`workplace_assistant`](https://github.com/NVIDIA-NeMo/Gym/tree/main/resources_servers/workplace_assistant)** — Multi-step tool calling in a workplace setting. Tools cover email, calendar, analytics, project management, and CRM toolkits across 5 databases. Verification matches the agent's final database state against ground-truth state.

**[`math_with_code`](https://github.com/NVIDIA-NeMo/Gym/tree/main/resources_servers/math_with_code)** — Mathematical reasoning with code execution. `execute_python()` runs in an isolated per-session process; state persists across steps. Verification extracts the boxed answer and compares against the expected result.

**[`example_session_state_mgmt`](https://github.com/NVIDIA-NeMo/Gym/tree/main/resources_servers/example_session_state_mgmt)** — Minimum stateful-environment reference. Demonstrates the `SESSION_ID_KEY` middleware pattern and the cookie-propagation contract.

For the full catalog grouped by domain, refer to [Benchmark Catalog](/latest/evaluate/benchmark-catalog).
