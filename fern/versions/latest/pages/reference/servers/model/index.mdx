Model servers are stateless LLM inference endpoints. They receive a conversation and return the model's next output (text, tool calls, or reasoning items) with no memory or orchestration. Every rollout has at least one Model server — the policy model under evaluation or training.

For the per-provider tested-version matrix, env.yaml fields, and known caveats, see [Provider Compatibility](/latest/reference/compatibility/providers). For the full architecture, see [Architecture](/latest/about/concepts/architecture).

## Base class

All model servers extend `SimpleResponsesAPIModel` (defined in `nemo_gym/base_responses_api_model.py`):

```python
class SimpleResponsesAPIModel(BaseResponsesAPIModel, SimpleServer):
    config: BaseResponsesAPIModelConfig

    @abstractmethod
    async def responses(self, body: NeMoGymResponseCreateParamsNonStreaming) -> NeMoGymResponse: ...

    @abstractmethod
    async def chat_completions(self, body: NeMoGymChatCompletionCreateParamsNonStreaming) -> NeMoGymChatCompletion: ...
```

`responses()` returns the OpenAI Responses-shaped output (with `output_text`, tool calls, and reasoning items). `chat_completions()` returns the OpenAI Chat Completions object. NeMo Gym buffers full responses; streaming endpoints are not exercised.

## Endpoints

| Path | Method | Purpose |
| --- | --- | --- |
| `/v1/responses` | POST | OpenAI Responses-shaped output. Default schema for all NeMo Gym rollouts. |
| `/v1/chat/completions` | POST | OpenAI Chat Completions output. |

The agent server only ever calls `/v1/responses`. Servers backed by chat-completions-only providers (e.g. `vllm_model` against OpenRouter) synthesize a Responses-shaped response via the bundled converter, so the agent doesn't need to know which underlying API the model server hit.

## Shipped server types

Six model server types ship under [`responses_api_models/`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models). Pick the one whose underlying API your provider exposes.

| Server | Underlying call | Use when |
| --- | --- | --- |
| [`openai_model`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models/openai_model) | `/v1/responses` (native) | OpenAI proper, build.nvidia.com NIMs that expose Responses, gpt-oss endpoints. |
| [`azure_openai_model`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models/azure_openai_model) | `/v1/chat/completions` (synthesized into Responses-shape) | Azure-hosted OpenAI deployments. |
| [`vllm_model`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models/vllm_model) | `/v1/chat/completions` | Self-hosted vLLM (remote), OpenAI-compatible gateways (OpenRouter, Together, Anyscale). |
| [`local_vllm_model`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models/local_vllm_model) | `/v1/chat/completions` (NeMo Gym manages vLLM lifecycle) | Run a local vLLM server inside the NeMo Gym process tree. |
| [`local_vllm_model_proxy`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models/local_vllm_model_proxy) | `/v1/chat/completions` (proxied) | Forward to a local vLLM whose lifecycle is owned by another process. |
| [`genrm_model`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_models/genrm_model) | `/v1/chat/completions` (managed local vLLM, custom roles) | Generative reward model with `response_1` / `response_2` / `principle` roles for pairwise comparison. |

Any `vllm_model`-family server can serve `/v1/responses` natively when its config sets `is_responses_native: true` (used today by gpt-oss-via-local-vLLM).

## Configuration

Model server configs are referenced by an agent's `model_server` field:

```yaml
policy_model:
  responses_api_models:
    openai_model:
      entrypoint: app.py
      openai_base_url: ${policy_base_url}
      openai_api_key: ${policy_api_key}
      openai_model: ${policy_model_name}
```

For full configuration syntax see [Configuration](/latest/reference/configuration).
