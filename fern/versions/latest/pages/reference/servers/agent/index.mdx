The Agent server is the orchestration component of an environment. It receives a task, calls the Model server for generation, dispatches any tool calls to the Resources server, and returns the verified reward. It does not run an LLM itself — all generation goes to the Model server, all tools and verification go to the Resources server.

For picking a harness for a new environment, see [Choose an Agent](/latest/get-started/choose-an-agent). For the high-level four-component model, see [Architecture](/latest/about/concepts/architecture).

## Base class

All agent harnesses extend `SimpleResponsesAPIAgent` (defined in `nemo_gym/base_responses_api_agent.py`):

```python
class SimpleResponsesAPIAgent(BaseResponsesAPIAgent, AggregateMetricsMixin, SimpleServer):
    config: BaseResponsesAPIAgentConfig

    @abstractmethod
    async def responses(self, body: NeMoGymResponseCreateParamsNonStreaming) -> NeMoGymResponse: ...

    @abstractmethod
    async def run(self, body: BaseRunRequest) -> BaseVerifyResponse: ...

    async def aggregate_metrics(self, body: AggregateMetricsRequest) -> AggregateMetrics: ...
```

`responses()` runs the model + tool-call loop and returns the final assistant response. `run()` runs a single end-to-end rollout (seed → respond → verify) and returns the verified reward. `aggregate_metrics()` runs the same `RewardProfiler` aggregation as the resources server by default; override to proxy.

## Endpoints

| Path | Method | Purpose |
| --- | --- | --- |
| `/v1/responses` | POST | Run the model + tool-call loop. Body: `NeMoGymResponseCreateParamsNonStreaming` (OpenAI Responses-compatible). Returns `NeMoGymResponse`. |
| `/run` | POST | Run a full rollout: seed the resources server, drive `/v1/responses`, then call `verify`. Body: `BaseRunRequest`. Returns `BaseVerifyResponse` with `reward`. |
| `/aggregate_metrics` | POST | Aggregate per-task metrics across rollouts. Body: `AggregateMetricsRequest`. Returns `AggregateMetrics`. |

Session middleware is wired automatically (`setup_session_middleware(app)`); cookies set on the first response carry session state across subsequent calls when the agent forwards them downstream.

## Cookie and token-ID propagation

For multi-turn or stateful environments, the agent must carry cookies and message-level token-IDs across calls. This is what makes per-rollout state work and what makes the resulting trajectories trainable.

- **Cookies** — pass `cookies=request.cookies` on every downstream call (resources server, model server). Capture the response cookies and forward them to the next call. See [Stateful environments](/latest/build-environments/supported-patterns/stateful) for the full pattern.
- **Token IDs** — `prompt_token_ids`, `generation_token_ids`, and `generation_log_probs` are attached by the model server to every output item. They ride along when you re-send the previous `output` list as the next turn's input — don't strip them. NeMo RL needs these for on-policy correction.

`simple_agent` and `proof_refinement_agent` are reference implementations of both contracts.

## Shipped harnesses

NeMo Gym ships 16 agent harnesses under [`responses_api_agents/`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents). Pick one when you wire up your YAML config; refer to [Choose an Agent](/latest/get-started/choose-an-agent) for the full comparison and decision criteria.

| Harness | Purpose |
| --- | --- |
| [`simple_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/simple_agent) | Default. Loops `/v1/responses` calls, dispatches tool calls back to the resources server, terminates on a final assistant message or `max_steps`. |
| [`non_executing_simple_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/non_executing_simple_agent) | Single forward pass; passes the model response straight to the verifier without executing tool calls. |
| [`proof_refinement_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/proof_refinement_agent) | Multi-turn correction loop with verifier error feedback. |
| [`gymnasium_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/gymnasium_agent) | Drives a Gymnasium-style `reset()` / `step()` loop. |
| [`tool_simulation_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/tool_simulation_agent) | Simulated tool execution (deterministic outputs for evals or SDG). |
| [`verifiers_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/verifiers_agent) | Bridges Prime Intellect's Verifiers library. |
| [`langgraph_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/langgraph_agent) | Adapter for LangGraph graphs. |
| [`harbor_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/harbor_agent) | Runs Harbor agents (e.g. `terminus-2`) inside Gym. |
| [`aviary_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/aviary_agent) | Bridges FutureHouse's Aviary framework. |
| [`stirrup_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/stirrup_agent) | Stirrup loop over GDPVal tasks. |
| [`mini_swe_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/mini_swe_agent) | Mini-SWE-Agent for SWE-Bench. |
| [`swe_agents`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/swe_agents) | Larger-footprint SWE-style harnesses. |
| [`browsecomp_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/browsecomp_agent) | Browse-and-answer harness for BrowseComp. |
| [`tau2`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/tau2) | τ²-bench customer-service harness. |
| [`cvdp_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/cvdp_agent) | Hardware-design harness (CVDP). |
| [`labbench2_vlm_agent`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/labbench2_vlm_agent) | Extends `simple_agent` with image/PDF inputs from disk. |

## Configuration

Agent configs reference a resources server, a model server, and one or more datasets:

```yaml
my_agent_instance:
  responses_api_agents:
    simple_agent:
      entrypoint: app.py
      resources_server:
        type: resources_servers
        name: my_resources_server
      model_server:
        type: responses_api_models
        name: policy_model
      datasets:
        - name: my_dataset
          type: train
          jsonl_fpath: path/to/data.jsonl
```

For full configuration syntax see [Configuration](/latest/reference/configuration).
