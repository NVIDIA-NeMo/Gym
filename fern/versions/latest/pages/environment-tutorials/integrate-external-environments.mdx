External projects use the word *environment* for different boundaries. Some expose reset, actions, observations, state, and reward independently of an agent. Others own the agent harness, sandbox, episode loop, and scoring as one end-to-end runner.

Choose the Gym integration seam from those runtime responsibilities, not from the upstream project's terminology.

## Map the upstream contract

Before implementing an adapter, identify:

- how tasks are discovered and identified;
- reset and per-episode lifecycle;
- action/tool and observation schemas;
- whether the framework owns the model/agent loop;
- termination conditions;
- whether rewards are terminal values, per-step deltas, cumulative values, or separate scorer results;
- verifier ownership;
- process, sandbox, and concurrency model;
- trajectory representation and model-call metadata;
- plugin/dependency installation.

Record any mapping that would be lossy.

## Choose the integration seam

### Resources Server adapter

Use a Resources Server when the external framework exposes environment behavior independently of the policy loop:

- reset creates isolated state;
- actions or tools advance that state;
- observations return to whichever agent is connected;
- reward or verification can be produced without requiring the upstream harness.

This preserves Gym's ability to pair the same environment with different agents and models.

[`resources_servers/openenv/`](https://github.com/NVIDIA-NeMo/Gym/tree/main/resources_servers/openenv) is the reference. It wraps OpenEnv reset/step behavior, creates one upstream environment instance per Gym session, exposes either MCP tools or `/step`, and translates upstream reward into `verify()`.

<Warning>
Define reward semantics explicitly. The OpenEnv adapter currently documents that summing observations is valid only for per-step deltas or terminal-only rewards; it is wrong for cumulative rewards.
</Warning>

### Agent Server adapter

Use an Agent Server when the framework owns a harness or an inseparable end-to-end episode runner:

- the framework decides when and how to call the model;
- its tools/sandbox are coupled to that harness;
- its own runner controls termination and scoring;
- splitting it would change benchmark semantics.

Wrap the runner in an async `/run` endpoint. Convert the Gym task row to upstream input, execute the episode, and translate trajectory, usage, and reward into `BaseVerifyResponse`.

Reference integrations:

- [`responses_api_agents/harbor_agent/`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/harbor_agent) wraps Harbor agents, environments, jobs, artifacts, and verifier results.
- [`responses_api_agents/verifiers_agent/`](https://github.com/NVIDIA-NeMo/Gym/tree/main/responses_api_agents/verifiers_agent) wraps Prime Intellect Verifiers environments whose `run()` owns the policy loop.

### Split adapter

Use both seams only when the framework has a stable environment API worth sharing independently and a separate framework-specific harness. Define one owner for reset/state/reward and one schema boundary between the two servers. Do not duplicate tools or score the same episode twice.

## Adapter requirements

### Configuration and dependencies

- Keep framework/environment selection in typed Gym config.
- Put server dependencies in its `requirements.txt` and pin versions needed for compatibility.
- Document plugin installation and use a separate preparation environment when dependency resolution conflicts with Gym.
- Pass reproducible runtime settings through Gym config. Keep secrets in supported local secret configuration.
- Support Linux.

### Schema and lifecycle

- Preserve upstream task IDs and deterministic reset inputs.
- Validate input and output conversion with Pydantic models.
- Create isolated upstream state per Gym session.
- Propagate Gym cookies across every stateful call.
- Define startup, readiness, cancellation, normal shutdown, forced termination, and artifact cleanup.
- Return model-caused tool errors to the agent when the upstream contract permits recovery; do not hide infrastructure failures as reward zero.

### Model calls and training metadata

Route model calls through the configured Gym Model Server when online training is required. Gym model responses carry:

- `prompt_token_ids`
- `generation_token_ids`
- `generation_log_probs`

Preserve these fields on assistant messages or Responses API output items when constructing the next multi-step request and when returning the completed trajectory. Dropping them can cause retokenization errors or make the rollout unusable for training.

Use `NeMoGymAsyncOpenAI` or a compatible upstream client that explicitly preserves these fields. See [OpenAI-compatible HTTP server and on-policy correction](/contribute/rl-framework-integration/openai-compatible-http-server-on-policy-correction).

### Async and scale

Keep `/run` and Resources Server endpoints async. Await Ray object references directly and avoid blocking the event loop. Use Gym's aiohttp-based clients and explicit cookie propagation.

See [Async patterns and performance](/infrastructure/async-patterns-and-performance) for HTTP client selection, third-party transport adapters, process lifecycle, and load testing.

## Validate the adapter

1. Contract-test reset, action/tool conversion, observations, termination, reward, and cleanup.
2. Exercise at least two upstream environments with different schemas when the framework supports them.
3. Run identical tasks upstream and through Gym.
4. Compare task counts, terminal state, trajectories, per-task rewards, and aggregate metrics.
5. Test concurrent sessions for state leakage.
6. Run a Gym evaluation smoke test.
7. If online RL is a goal, verify token IDs and log probabilities with the target training framework.
8. Document supported capabilities and known incompatibilities.

A reusable framework adapter is successful when another compatible upstream environment can be enabled primarily through configuration and data rather than a new Python wrapper.
