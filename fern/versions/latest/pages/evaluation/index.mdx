NeMo Gym evaluation is environment-native: the same dataset, resources server, verifier, and rollout machinery can support model comparison, harness comparison, benchmark development, ablation studies, and training-data analysis.

## Gym Evaluation Loop

1. Choose a task set and resources server.
2. Choose the model server and agent harness.
3. Collect rollouts with fixed sampling settings and an explicit repeat count.
4. Verify each rollout and compute aggregate metrics.
5. Compare against a baseline or ablation.
6. Inspect failures, sometimes-pass tasks, and missing rows.
7. Decide whether the next action is model training, harness work, verifier repair, benchmark cleanup, or data filtering.

The most important rule is to vary one thing at a time for causal comparison. Changing the model, harness, prompt, verifier, and dataset together can still produce a useful score as a release gate, but it cannot explain what caused the difference.

## Models

Model evaluation asks: *with the same environment and harness, which model performs better?*

Hold these fixed:

- Dataset and task split
- Resources server and verifier version
- Agent harness
- Prompt configuration
- Sampling settings, max tokens, and tool budget
- Repeat count

Then vary the model server. This can compare hosted models, self-hosted vLLM models, local development checkpoints, or a policy model before and after training.

For model comparisons, record more than the headline reward. Capture token usage, latency, timeout rate, malformed outputs, tool-call validity, and verifier messages when available. A model that gains accuracy by tripling cost or timing out on harder tasks may not be the better deployment choice.

<Tip>

Use the [Model Server](/model-server) docs to keep model-provider differences behind the Responses API boundary.

</Tip>

## Harnesses

Harness evaluation asks: *with the same model and environment, which agent loop performs better?*

Hold the model fixed, then vary the agent server implementation, system prompt, tool policy, context management, retry logic, planning strategy, or stop criteria. This isolates orchestration quality from raw model capability.

Harness changes often affect metrics beyond reward:

- Steps or turns per task
- Tool calls per successful task
- Retry and recovery behavior
- Context length growth
- Sandbox/runtime failures
- Token and latency cost

NeMo Gym treats an agent as model plus harness. The model server stays stateless; the agent server owns the loop that calls the model, routes tool calls, manages conversation state, and asks the resources server to verify the final attempt.

## Benchmarks

A benchmark is an environment configured for repeatable comparison. It needs a task set, resources server, verifier, expected runtime, documented dependencies, and known metrics. Public benchmarks are useful for community comparison; custom benchmarks are usually better for product-specific behavior.

When choosing or building a benchmark, check:

- **Task fit**: Does the task distribution match the target capability?
- **Verifier fidelity**: Does reward measure the target behavior, or just a proxy?
- **Model compatibility**: Can the model endpoint support the required input/output mode, tool calls, images, long context, or structured output?
- **Harness fit**: Does the harness expose the tools and interaction pattern required by the benchmark?
- **Variance**: Are repeats stable enough to distinguish real changes from noise?
- **Reproducibility**: Can another run use the same data, code, config, and model reference?

For integrated external benchmarks, first reproduce the original benchmark's reported numbers outside Gym when possible. Then integrate it into Gym and rerun against the same model set. This separates benchmark reproduction issues from Gym integration issues.

## Ablations And Iterations

Evaluation is most useful when it drives a controlled iteration loop. Treat each proposed improvement as an ablation:

| Question | Hold Constant | Vary |
|---|---|---|
| Did a new checkpoint improve capability? | Harness, data, verifier, sampling | Model server |
| Did a harness change help? | Model, data, verifier, sampling | Agent server or prompt |
| Did verifier repair change the metric? | Model, harness, data | Resources server verifier |
| Did data filtering help training? | Model family, harness, verifier | Training data slice |
| Did a new tool help enough to justify cost? | Model, task set, verifier | Harness tool policy |

Use multiple repeats for stochastic or multi-step tasks. The highest-signal rows are often sometimes-pass tasks, where the same task succeeds in some rollouts and fails in others. They reveal whether the failure is a knowledge gap, unreliable behavior, harness issue, verifier issue, infrastructure problem, or data artifact.

Keep the iteration record with the artifacts:

- Git SHAs for environment, agent, and model-server code
- Config files and prompt configs
- Model IDs or checkpoint references
- Sampling settings and repeat count
- Rollout JSONL paths
- Aggregate metrics and reward profiles
- Notes on missing rows, timeouts, or known flaky infrastructure
