# BBH Hypotest in `sandbox_agent`

This configuration runs OpenCode through the generic `sandbox_agent`. The
`bbh_hypotest` resources server seeds each Hypotest capsule on shared storage,
returns a trusted workspace handoff, and later replays the retrieved analysis
artifact before rubric scoring.

OpenCode receives the rollout-prefixed Gym model URL directly. No separate
model bridge or `Handler` is required when that endpoint is reachable from the
sandbox. The model server buffers each turn under the run token, injects exact
prior token IDs into later OpenCode requests, and returns the reconstructed
monotonic trajectory to the agent before verification. This avoids retokenizing
multi-turn assistant history and does not use observability capture records.
The OpenCode build agent is allowed 128 agentic iterations and is instructed to
create `analysis.py` before lengthy exploration, so a normal multi-turn context
does not get mistaken for a rollout failure and an early exit still has a
replayable artifact target.

Required environment variables are `BBH_PROBLEM_JSONL`, `BBH_CAPSULE_DIR`,
`BBH_OPENSANDBOX_IMAGE`, `OPENSANDBOX_DOMAIN`, `OPENSANDBOX_API_KEY`, and
`OPENAI_API_KEY`. `BBH_SANDBOX_GYM_SOURCE` may point at a prebuilt Gym source
archive; it defaults to an archive generated by `sandbox_agent` at startup.
The `bbh_hypotest` component requirements pin the compatible Hypotest revision.
`BBH_OPENCODE_CONTEXT` defaults to 79,999 when vLLM uses an 80,000-token model
length; the one-token headroom accounts for OpenCode's inclusive input/output
boundary without reducing the policy training sequence length.
