CUBE OSWorld — request flow

Two views of the same stack: how cube_agent orchestrates policy + environment during one rollout, and what the cube resources server owns (process lifecycle, QEMU, HTTP surface). Rollout collection talks to the head server, which forwards to child servers.

1. cube_agent point of view

One POST /run (from the head, on behalf of ng_collect_rollouts) runs this loop. The agent holds env_id and growing responses_create_params (multimodal context).

Not shown: after a batch, the collector may call POST /aggregate_metrics on the agent with stripped results (separate from a single /run).

2. Cube resources server responsibilities

One FastAPI app per child process. eager_benchmark_init + optional VM warmup run before uvicorn accepts traffic (when enabled in YAML). 

flowchart TB
  ETC[eager load benchmark]
  WU[optional VM warmup]
  UP[uvicorn up]
  MAP[env_id to Task]
  RW[reward totals]

  ETC --> WU
  WU --> UP
  UP --> SEED[seed_session]
  UP --> STP[step]
  UP --> CLS[close]
  UP --> VFY[verify]

  SEED --> MAP
  STP --> MAP
  STP --> RW
  CLS --> MAP
  VFY --> RW

Each /seed_session creates a new OSWorldTask and VM; warmup only pre-touches cold paths at ng_run time. Concurrent rollouts mean multiple env_ids in env_id_to_task until closed—bounded by host RAM/CPU, not a fixed pool in this codebase.
