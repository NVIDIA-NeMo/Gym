Cube resources server ↔ CUBE / cube-harness primitives

How NeMo-Gym’s resources_servers/cube types sit on top of packages from the cube-harness repo (e.g. osworld-cube git install), the shared cube core API, and cube_computer_tool. Names match Python import paths used in server.py, environments/osworld/, and adapters.py.

1. Type relationships (who holds what)

High-level wiring only; see Python under server.py, environments/osworld/, adapters.py for full types. 

flowchart TB
  CFG[CubeResourcesServerConfig]
  SRV[CubeResourcesServer]
  BOOT[ensure_osworld_tasks]
  BENCH[OSWorldBenchmark]
  TLIST[task config list]
  LIVE[env_id to live Task]
  ADP[adapters module]
  CUBE[Action Observation from cube.core]

  CFG --> SRV
  BOOT --> BENCH
  BENCH --> TLIST
  BENCH --> SRV
  SRV --> LIVE
  SRV --> ADP
  ADP --> CUBE

2. Bootstrap vs HTTP: data flow

flowchart TB
  YML[YAML extra fields]
  EOT[bootstrap]
  BC[OSWorldBenchmark]
  ST[server state lists]
  SD[HTTP seed_session]
  SP[HTTP step]
  CL[HTTP close]
  VM[env_id task map]

  YML --> EOT
  EOT --> BC
  BC --> ST
  SD --> ST
  SD --> VM
  SP --> VM
  CL --> VM

Package origins: osworld-cube is installed from git+https://github.com/The-AI-Alliance/cube-harness.git#subdirectory=cubes/osworld-cube (see resources_servers/cube/requirements.txt). cube and cube_computer_tool are pulled in as dependencies of that stack. NeMo-Gym does not vendor cube-harness; it wires YAML → benchmark → Task and translates CUBE types to OpenAI / NeMo-Gym HTTP shapes.
