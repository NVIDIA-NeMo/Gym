# Synthetic Tool-Use Generation

The generation workflow builds customer-service domains, policies, tool schemas, and customer scenarios before any
Gym rollout starts. Each stage is independently resumable and exchanges files through one run directory.

## Stages

### Domain Generation

The domain stage sends an initial brainstorming prompt to the configured domain model, parses the returned JSON
candidates, and sends a follow-up request that excludes names already generated by the first request. Candidate names
are normalized before deduplication.

The stage writes:

- `domains.raw.jsonl`: every parsed candidate, including rejected duplicates
- `domains.accepted.jsonl`: candidates selected for downstream generation
- `domains/<index>/domain.json`: one durable artifact per accepted domain
- `attempts/domains/*.json`: provider responses and parsing diagnostics

### Policy And Tool Generation

The policy/tool stage loads either the `general` or `proactive` profile. For each accepted domain it:

1. Generates a policy using shuffled policy/tool references.
2. Generates JSONL tool schemas for that policy.
3. Optionally refines the policy and tools.
4. Rejects empty policies, prompt-template leakage, malformed JSON Schema, and duplicate tool names.
5. Optionally runs cohesion votes and comparisons against reference policy/tool pairs.

Accepted artifacts are written as `policy.md`, `tools.jsonl`, and `quality_report.json`. Every semantic attempt keeps
its raw provider responses under the domain's `attempts/policy_tools` directory.

### Scenario Generation

The scenario stage combines each accepted policy with a schedule of inside-policy and outside-policy requests. It
renders the system and user prompts, asks the scenario model for a structured scenario collection, validates every
scenario, and removes duplicates across concurrent requests.

Accepted scenarios are split into JSONL files under `scenarios/<model>/`. The domain quality report records accepted
counts, failed requests, and the inside/outside-policy balance.

### Validation And Materialization

Validation checks that each domain has a usable policy, unique tools with valid parameter and return schemas, and at
least one complete scenario. Materialization converts valid domain bundles into Gym input rows for the synthetic
tool-use agent and simulation resource server.

## Configuration

Two complete configs are checked in:

- [`general.yaml`](../configs/general.yaml)
- [`proactive.yaml`](../configs/proactive.yaml)

Each config selects four model roles:

- `domain_model`
- `policy_tools_model`
- `judge_model`
- `scenario_model`

Model roles configure the model name, endpoint, API-key environment variable, timeout, provider retry policy,
concurrency, and sampling arguments. Stage blocks configure request counts, semantic attempts, refinement, judging,
scope balance, and scenario file size.

## Running

Run every stage and materialize a dataset:

```bash
python resources_servers/synthetic_tool_use/scripts/generate_synthetic_tool_use_seeds.py \
  --config resources_servers/synthetic_tool_use/configs/general.yaml \
  --resume \
  all \
  --dataset-name synthetic_tool_use_general \
  --output-path /tmp/synthetic_tool_use_general.jsonl
```

Replace `all` with `domains`, `policies`, `scenarios`, `validate`, or `materialize` to run a single operation against
the same run directory. `--domain-start` is inclusive and `--domain-end` is exclusive for partitioned policy/tool and
scenario workers.

## Resume And Diagnostics

`run_manifest.json` tracks run configuration, asset hashes, domain identities, stage status, attempt counts, and
scenario counts. With `--resume`, a completed stage is skipped only when its expected artifacts still load and pass
validation.

Provider retries handle transient HTTP and connection failures. Semantic attempts handle responses that arrive but
cannot be parsed, validated, or accepted by the quality gates. Both kinds of attempts are recorded without exposing
API credentials or endpoint URLs in the manifest.
